# TLS / HTTPS 的核心机制

定位：这部分不是前端安全常识，而是后端工程师必须掌握的链路安全基础。它直接关系到证书配置、网关终止 TLS、服务间 mTLS、握手延迟和安全排障。

## 1. HTTPS 到底比 HTTP 多了什么

最常见的错误回答是“HTTPS 就是 HTTP 加密版”。这个说法不算错，但太浅。

更准确地说：

- HTTP 是应用层协议，负责定义请求方法、头、状态码、消息体等语义。
- TLS 是传输层之上的安全层，负责加密、完整性校验和身份认证。
- HTTPS 本质上是“HTTP over TLS”。

所以 HTTPS 并没有改写 HTTP 的业务语义，而是在 HTTP 下面插了一层安全信道。

## 2. TLS 解决的三个核心问题

### 2.1 机密性

- 防止中间链路直接看到明文请求和响应。

### 2.2 完整性

- 防止数据在传输中被篡改而双方感知不到。

### 2.3 身份认证

- 客户端要能验证自己连到的确实是目标服务，而不是伪造站点。

高级工程师回答这类题时，最好明确指出：

- 仅有加密不够，如果没有身份认证，仍然可能被中间人拿自己的密钥把流量“合法加密”后转发。

## 3. 为什么不能直接用非对称加密传全部数据

很多人以为 TLS 全程就是 RSA 或 ECC 在硬加密业务数据。实际上不是。

原因很简单：

- 非对称加密计算成本高。
- 直接用它加密大流量数据，性能开销太大。

TLS 的常见套路是：

1. 握手阶段用非对称密码学或密钥交换算法建立信任与共享秘密。
2. 业务数据阶段改用对称加密传输。

所以非对称加密主要用来解决“怎么安全地协商密钥”，而不是高吞吐地承载全部业务报文。

## 4. TLS 握手大致做了什么

忽略版本差异后，可以把 TLS 握手理解为完成了几件事：

1. 协商协议版本和密码套件。
2. 服务端出示证书。
3. 客户端验证证书是否合法、是否信任、域名是否匹配。
4. 双方协商出后续会话使用的对称密钥。
5. 后续应用数据使用对称加密和完整性保护机制传输。

面试时不一定要背每个握手报文名，但必须说清楚“证书验证”和“会话密钥协商”是两件关键事。

## 5. 证书到底证明了什么

证书不是“证明这个网站绝对安全”，它证明的是：

- 某个公钥属于某个身份主体。
- 这个绑定关系由受信任 CA 或受信任体系签名背书。

客户端信任的是：

- 本地信任链里的 CA。
- 证书签名链条。
- 域名匹配规则。
- 证书有效期、吊销状态等。

所以证书的核心价值是“把公钥和身份绑定起来”，从而避免你拿到一个不知道属于谁的公钥。

## 6. 中间人攻击为什么会被 TLS 抵抗

如果没有证书体系，中间人完全可以拦截请求并把自己的公钥发给客户端，客户端若不校验身份，就会误以为这是服务端公钥。

TLS 抵抗中间人攻击的关键不是“链路被加密了”，而是：

- 客户端会验证服务端证书是不是由受信任 CA 签发。
- 还会校验证书中的域名是否和访问目标匹配。

中间人即使能篡改流量，也很难伪造一张被客户端信任且域名匹配的证书。

所以别把“加密”和“认证”混为一谈。真正抵御中间人的核心是认证链条。

## 7. TLS 1.2 和 TLS 1.3 你至少要知道什么

后端面试一般不要求你背完整 RFC，但至少要知道：

- TLS 1.3 对握手流程做了简化，减少往返延迟。
- 移除了一批不安全或过时的算法与模式。
- 更强调前向安全等现代安全实践。

如果服务对首包延迟敏感，TLS 版本和握手轮次会直接影响链路时延。

## 8. 什么是前向安全（Forward Secrecy）

前向安全的意思是：

- 即使服务端长期私钥未来泄漏，也不应该让过去抓下来的所有历史会话流量都被统一解密。

这是现代 TLS 设计里非常重要的安全目标。它要求会话密钥不能简单等价于“长期私钥直接决定的一把固定钥匙”。

工程上不要求你现场推导密码学细节，但要知道这是现代密钥交换设计的重要价值。

## 9. 为什么 HTTPS 会更慢

HTTPS 的额外开销主要来自：

1. TLS 握手引入额外 RTT。
2. 密钥交换和证书验证有 CPU 成本。
3. 会话建立前不能直接裸传应用数据。

但是这不代表 HTTPS 一定“很慢”。

工程上通常会通过：

- TLS 1.3
- 会话复用
- 连接复用
- 长连接
- 反向代理统一终止 TLS

来降低额外成本。

## 10. TLS 终止（TLS termination）是什么意思

现实系统里，TLS 不一定终止在最终业务进程。

常见做法是：

- 在负载均衡器、网关、Ingress、Nginx、Envoy 处完成 TLS 握手和解密。
- 后面的内部链路再决定是否继续走明文或继续走 mTLS。

这叫 TLS termination。它带来的工程权衡包括：

- 统一证书管理更方便。
- 业务服务减轻 TLS 握手和证书管理负担。
- 但内部链路如果改成明文，需要明确边界与风险。

## 11. 什么是 mTLS

普通 HTTPS 通常是服务端向客户端证明身份。

mTLS 是双向认证：

- 服务端给客户端看证书。
- 客户端也给服务端看证书。

这在服务间通信、零信任网络、内部高安全场景很常见。它解决的不只是“加密”，更重要的是双向身份确认。

## 12. 面试里怎么答得更像高级工程师

如果面试官问“HTTPS 为什么安全”，不要只说“因为加密”。更好的结构是：

1. 先定义 HTTPS 是 HTTP over TLS。
2. 再说 TLS 解决机密性、完整性、身份认证三个问题。
3. 解释证书如何绑定公钥与身份。
4. 强调 TLS 抵御中间人的关键在认证链条，而不只是加密。
5. 最后补充 TLS 1.3、握手成本、TLS termination、mTLS 等工程实践。

这样回答才真正体现出后端工程师对协议和系统落地的理解。

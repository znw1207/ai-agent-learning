# `select`、`poll` 和 `epoll`

定位：这是后端网络编程里最典型的“从会写阻塞服务到理解高并发事件驱动”的分水岭知识点。高级工程师不一定每天手写这几个 API，但必须理解它们为什么存在、各自的成本模型是什么，以及为什么现代 Linux 高并发服务器更偏向 `epoll`。

## 1. 它们在解决什么问题

如果你为每个连接创建一个线程，然后在线程里阻塞 `read()`，实现上很直观，但连接一多就会出现：

- 线程数量过多
- 上下文切换开销高
- 内存占用大
- 大量线程只是“等 IO”而不是“干活”

所以问题不是“如何更快地读一个 socket”，而是：

- 如何让一个线程高效地等待很多 fd 的状态变化

`select`、`poll`、`epoll` 都是在解决这个问题，也就是 IO 多路复用问题。

## 2. `select` 的工作方式

`select` 的基本思路是：

1. 应用层准备几个 fd 集合，比如读集合、写集合、异常集合。
2. 调用 `select()` 把这些集合交给内核。
3. 内核阻塞等待，直到某些 fd 就绪。
4. 返回后应用层再遍历整个集合，找出哪些 fd 真正可读可写。

### 2.1 `select` 的优点

- 接口历史悠久，兼容性好。
- 模型直观，适合教学和小规模连接管理。

### 2.2 `select` 的问题

- fd 数量通常受 `FD_SETSIZE` 限制。
- 每次调用都要把整批 fd 集合从用户态拷到内核。
- 返回后应用层还要线性扫描整套集合。

所以 `select` 的瓶颈不在于“它不能工作”，而在于连接规模一大，重复拷贝和全量扫描成本很明显。

## 3. `poll` 的工作方式

`poll` 的思路和 `select` 类似，但表示方式变了：

- 应用层传的是 `struct pollfd` 数组，而不是位图集合。
- 每个元素里带 fd 和关心的事件类型。

### 3.1 `poll` 相比 `select` 的改进

- 不再受 `FD_SETSIZE` 这种固定集合大小的明显限制。
- 接口表达能力更自然一些。

### 3.2 `poll` 仍然存在的问题

- 每次调用仍要把关注的 fd 列表交给内核。
- 返回后应用层仍要遍历整个数组找就绪 fd。

所以 `poll` 可以理解成：

- 比 `select` 更灵活，但没有从根上摆脱“每次调用都全量扫描”的成本模型。

## 4. `epoll` 的工作方式

`epoll` 的关键变化不只是“新 API”，而是把两件事拆开了：

1. 注册关注哪些 fd，用 `epoll_ctl()`。
2. 等待哪些 fd 真正就绪，用 `epoll_wait()`。

这意味着：

- 应用不需要每次等待时都重新把完整 fd 集合交给内核。
- 内核维护关注列表。
- 返回时直接给你“已经就绪的事件列表”。

这比 `select`/`poll` 的全量提交 + 全量扫描模型更适合大规模连接场景。

## 5. 三者最核心的差别该怎么记

如果用一句话概括：

- `select`：位图 + 全量拷贝 + 全量扫描 + 有 fd 数量上限问题。
- `poll`：数组 + 全量拷贝 + 全量扫描 + 没有那么死的 fd 上限。
- `epoll`：注册和等待分离，返回就绪事件，更适合大规模连接。

面试时不要只答“`epoll` 更快”，这种说法太空。更准确的说法是：

- `epoll` 更适合大量 fd、低活跃比例的事件驱动场景。

## 6. 为什么 `epoll` 更适合高并发服务器

真实互联网服务的常见特征是：

- 连接数很多
- 但任意时刻真正活跃的连接只是一部分

这时如果每次都对全部 fd 做扫描，成本就很浪费。`epoll` 的优势就在于：

- 由内核维护关注集合
- 应用只处理真正就绪的事件
- 减少无意义扫描和重复集合传递

所以它特别适合：

- 网关
- 代理
- IM 服务
- 长连接服务
- 高并发 RPC 服务

## 7. `epoll` 的 LT 和 ET 该怎么理解

### 7.1 LT：Level Triggered

- 只要 fd 仍然可读或可写，就可能持续收到通知。
- 编程相对简单，不容易漏事件。

### 7.2 ET：Edge Triggered

- 更强调状态变化，只在从“不可用”变成“可用”时重点通知。
- 程序通常要一次性尽量读到 `EAGAIN`。

高级工程师应该知道：

- ET 不是天然更高级。
- ET 对实现正确性要求更高。
- 如果读写循环没处理好，很容易丢事件或造成连接卡死。

## 8. 为什么这几个机制常和非阻塞 socket 一起出现

如果 fd 是阻塞的，那么即使你通过 `select`/`poll`/`epoll` 知道某个 fd 有事件，后续一次阻塞 `read()` 或 `write()` 仍可能把线程卡住，拖慢整个事件循环。

所以成熟事件驱动模型通常是：

- fd 设为非阻塞
- 先等待事件
- 事件就绪后循环读写直到 `EAGAIN`

这尤其是 `epoll` 高频使用场景的标准套路。

## 9. 面试里怎么答得更像高级工程师

如果被问“`select`、`poll`、`epoll` 区别”，建议按这个结构答：

1. 先说三者都在解决“一个线程等很多 fd”这个 IO 多路复用问题。
2. 再说 `select` 是位图模型，有上限且每次全量扫描。
3. 说 `poll` 更灵活，但本质仍是全量提交和扫描。
4. 说 `epoll` 把注册和等待分离，返回就绪事件，更适合大规模连接。
5. 最后补充非阻塞 fd、LT/ET、慢 handler 仍会拖垮事件循环。

这样就不是只背 API，而是在讲成本模型和系统设计。

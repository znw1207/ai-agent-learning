# RPC 与服务治理

定位：高级后端面试里，网络问题通常不会停在 TCP 或 HTTP 层，最后都会落到服务间调用怎么治理。RPC 是协议问题，更是稳定性、可观测性和组织协作问题。

## 1. RPC 到底在解决什么问题

RPC 的核心目标是：

- 让调用远程服务像调用本地函数一样更方便，但又不能掩盖网络调用的真实成本和失败语义。

高级工程师应该清楚：

- RPC 不是“把网络消失”，而是“把网络调用标准化”。

它通常解决：

- 接口契约定义。
- 序列化和反序列化。
- 连接管理。
- 服务发现。
- 超时、重试、负载均衡、熔断、鉴权、观测等治理能力。

## 2. 为什么很多场景用 RPC 而不是直接 HTTP REST

不是因为 HTTP 不行，而是因为 RPC 在服务间调用里通常更强调：

- 明确契约
- 高效序列化
- 统一客户端与中间件能力
- 更自然的双向流和多路复用

像 gRPC 这样的方案，本质上还是跑在 HTTP/2 之上，但对服务间通信做了更强的工程抽象。

## 3. gRPC 为什么常被说更快

更准确的说法不是“gRPC 天生更快”，而是它常见场景下更容易获得更好的效率：

- 基于 HTTP/2，多路复用更好。
- Protobuf 等二进制序列化更紧凑。
- 代码生成让契约更统一。
- 流式通信支持更自然。

但高级工程师不会把它神化。真正性能还取决于：

- 数据结构大小
- 压缩与序列化成本
- 下游处理时长
- 代理与网关支持
- 观测与调试成本

## 4. 服务发现为什么是 RPC 的关键部分

调用一个服务，不只是知道一个域名那么简单，还涉及：

- 当前有哪些实例可用。
- 哪些实例健康。
- 如何做负载均衡。
- 实例上下线如何感知。

所以服务发现和负载均衡通常是 RPC 客户端能力的重要组成部分。

## 5. 超时传播为什么重要

如果入口请求总预算只有 2 秒，而中间每一层都各自默认等 2 秒，那么链路一定会在尾部放大。

所以成熟系统会做 timeout propagation，也就是：

- 上游把剩余时间预算传给下游。
- 下游不能随意用比上游更松的默认超时。

这能避免整条调用树对时间预算没有统一认知。

## 6. 重试应该放在哪一层

这是很经典的面试点。标准工程思路不是“每层都配一点”，而是：

- 选择最有上下文的一层做主要重试。
- 其他层尽量避免叠加放大。

例如：

- 网关层可能适合处理短暂网络错误。
- 业务层更了解请求是否幂等、是否可重试。
- 存储层或底层 SDK 若也各自重试，很容易产生倍数放大。

## 7. 熔断、限流、隔离为什么必须一起看

### 7.1 限流

- 控制入口流量，不让系统超载。

### 7.2 熔断

- 当下游持续异常时，快速失败，避免继续浪费资源。

### 7.3 隔离

- 把不同依赖、不同租户、不同调用类型的资源分开，避免相互拖死。

它们共同目标都是：

- 防止局部故障扩散成系统故障。

## 8. 为什么链路追踪和请求染色很重要

RPC 系统一旦跨多服务，单点日志很快失效。

所以需要：

- trace id 串联整条调用链。
- span 记录每跳耗时。
- 请求染色或流量标识辅助灰度、压测、问题隔离。

高级工程师谈服务治理时，如果完全不提观测能力，说明只关注“怎么调用”，没关注“出问题怎么找”。

## 9. 面试里更像高级工程师的答法

如果被问“RPC 和 HTTP 的区别”或“如何做服务治理”，建议按这个结构答：

1. 先说 RPC 是把远程调用标准化，而不是把网络成本隐藏掉。
2. 再说契约、序列化、连接管理、服务发现是其核心能力。
3. 解释超时传播、重试层次和幂等约束。
4. 再补限流、熔断、隔离和链路追踪。
5. 最后强调治理目标是防放大、可观测、可恢复，而不只是“调用成功”。

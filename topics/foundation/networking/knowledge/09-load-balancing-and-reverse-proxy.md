# 负载均衡与反向代理

定位：这是服务真正接流量时的入口层基础。高级工程师不仅要知道四层和七层的定义，还要知道它们如何影响转发能力、可观测性、协议特性和故障放大方式。

## 1. 什么是负载均衡，什么是反向代理

### 1.1 负载均衡

- 核心目标是把流量分散到多个后端实例，提升可用性、吞吐和扩展性。
- 它强调的是“流量分发”。

### 1.2 反向代理

- 代理位于服务端前面，客户端只看到代理，不直接看到真实后端。
- 它不仅能转发流量，还可以做 TLS 终止、鉴权、缓存、限流、压缩、路由、观测等。

工程上两者常常重叠。例如 Nginx/Envoy 既是反向代理，也常承担负载均衡职责。

## 2. 四层负载均衡和七层负载均衡的区别

### 2.1 四层负载均衡

- 工作在传输层，主要看 IP、端口、协议等信息。
- 不理解 HTTP 路径、Header、Cookie、Method 等应用层语义。
- 转发开销通常更低，性能高，适合通用 TCP/UDP 流量。

### 2.2 七层负载均衡

- 工作在应用层，能够理解 HTTP/HTTPS、gRPC 等上层协议。
- 可以按 Host、Path、Header、Cookie、方法、权重等更细粒度转发。
- 能做更丰富的策略，但也意味着更复杂、更重。

一句话区分：

- 四层更偏“快和通用”。
- 七层更偏“懂协议、策略强”。

## 3. 为什么很多系统会四层 + 七层叠加

真实系统里常见架构不是二选一，而是组合：

- 外层四层负载均衡先把连接分散到入口代理集群。
- 内层七层代理再做 Host/Path/服务级路由、鉴权、限流和灰度。

这样做的好处是：

- 接入层更通用、更抗压。
- 应用层策略更灵活。
- 各层职责更清晰。

## 4. 反向代理为什么能扛高并发

反向代理能抗高并发，不是因为它“只是转发所以轻”，而是因为它通常做了几件工程上关键的事：

1. 连接复用，减少后端频繁建连成本。
2. 事件驱动和 IO 多路复用，少量线程管理大量连接。
3. TLS 终止集中化，避免每个业务进程都重复做握手成本。
4. 统一做限流、缓存、压缩、熔断等保护，减轻后端压力。

所以 Nginx/Envoy 这类组件的价值不只是“转发”，而是把高并发入口处理中的共性能力集中化。

## 5. 常见负载均衡策略应该怎么理解

常见策略包括：

- 轮询
- 加权轮询
- 最少连接
- 一致性哈希
- 按延迟或健康状态动态选择

面试时不要只背名称，更要知道适用场景：

- 一致性哈希适合会话粘性、缓存命中场景。
- 最少连接适合连接生命周期较长、负载不均匀的情况。
- 简单轮询适合后端能力比较接近、请求模式较均匀的场景。

## 6. 会话保持（session affinity）为什么既有用又危险

会话保持的优点是：

- 某些有本地缓存、会话状态、本地热数据的服务更容易命中已有状态。

问题是：

- 容易导致热点实例。
- 缩容和故障切换更复杂。
- 会掩盖状态没做好外部化的问题。

所以高级工程师面对“是否做会话粘性”时，应该先问业务状态是不是必须粘住，而不是默认开启。

## 7. 反向代理层为什么也会放大故障

入口代理不是天然安全垫，它自己也可能成为故障放大器：

- 后端变慢导致代理连接池占满。
- 代理超时设置不合理，导致请求大量堆积。
- 代理层重试叠加业务层重试，引发放大。
- 单机连接数、文件描述符、内存 buffer 被耗尽。

所以你不能把代理看成“无副作用的透明中转”。它本身就是系统的一部分，也要治理超时、连接池、重试和限流。

## 8. 为什么连接复用会影响后端表现

代理层若做好连接复用，可以显著减少后端频繁三次握手、TLS 握手和短连接抖动。但如果配置不合理，也会带来问题：

- 单连接承载过多请求，异常影响范围变大。
- 长连接空闲管理不当，连接泄漏或空转占资源。
- 上游和下游 keepalive 参数不匹配，产生大量 reset 或半开连接。

所以连接复用不是“开了就完”，而是需要结合请求模式、超时和后端能力一起调。

## 9. 面试里更像高级工程师的答法

如果被问“四层和七层负载均衡有什么区别”，建议按这个结构答：

1. 先说四层基于 IP/端口，七层理解应用层协议。
2. 再说四层性能更通用，七层策略更灵活。
3. 说明现实中常常是叠加，而不是替代。
4. 补充代理层可做 TLS、限流、缓存、路由、观测。
5. 最后强调代理层自身也会成为性能和故障放大点。

这样就从“定义题”进到了“系统题”。

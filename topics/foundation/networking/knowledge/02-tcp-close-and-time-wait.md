# TCP 断连、四次挥手与 TIME_WAIT

定位：这部分知识直接影响你对线上连接泄漏、端口耗尽、短连接性能、代理层异常和服务优雅关闭的理解。

## 1. 为什么断连不是三次，而通常是四次

TCP 是全双工协议，双方的发送和接收是相对独立的。

这意味着：

- 我可以先告诉对方“我这边没有数据要发了”。
- 但对方此时可能还有数据没发完。
- 所以连接关闭通常需要分别关闭两个方向。

标准过程通常是四次挥手：

1. 主动关闭方发送 `FIN`，表示“我这边不再发送数据”。
2. 被动关闭方回复 `ACK`，表示“我知道了”。
3. 被动关闭方处理完剩余数据后，再发送自己的 `FIN`。
4. 主动关闭方回复最后一个 `ACK`。

所以四次挥手的本质不是“协议故意设计得复杂”，而是因为 TCP 连接的两个方向需要分别关闭。

## 2. 半关闭（half-close）是什么意思

收到对方的 `FIN`，不代表这个连接马上完全不可用了。

- 它只表示对方不再发送数据给你。
- 你仍然可以继续向对方发送数据，直到你也关闭发送方向。

这就是半关闭。高级工程师回答这类题时，最好顺手提一句：

- 关闭连接不是单一瞬间动作，而是双方发送方向分别结束的过程。

## 3. `TIME_WAIT` 为什么存在

`TIME_WAIT` 常出现在主动关闭连接的一方。它的存在主要有两个目的。

### 3.1 保证最后一个 ACK 能被对方收到

- 如果主动关闭方发出的最后一个 ACK 丢了，被动关闭方会重发 `FIN`。
- 主动关闭方必须还保留足够状态，才能重新回复 ACK。

### 3.2 防止旧连接的延迟报文污染新连接

- 网络里可能仍然残留旧连接的报文。
- 如果连接四元组被太快复用，这些旧报文可能被误认为是新连接的数据。
- `TIME_WAIT` 给旧报文一个自然过期窗口。

所以 `TIME_WAIT` 不是“浪费资源的垃圾状态”，而是 TCP 为可靠关闭和报文隔离付出的必要成本。

## 4. 为什么高并发短连接服务容易遇到 `TIME_WAIT`

如果服务大量使用短连接，并且经常由服务端主动关闭连接，那么服务端会积累大量 `TIME_WAIT`。

典型后果包括：

- 本地临时端口耗尽。
- 文件描述符和连接状态过多，影响系统资源。
- 监控上看到连接数异常升高，但不一定代表真正业务请求变多。

这类问题不能靠简单“关掉 `TIME_WAIT`”解决。更合理的思路是：

- 优先使用连接复用和长连接。
- 优化代理层和服务端的关闭策略。
- 必要时结合内核参数做谨慎调优，但前提是理解副作用。

## 5. `CLOSE_WAIT` 为什么常被看作代码问题

`CLOSE_WAIT` 一般出现在被动关闭的一方，表示：

- 对方已经发来了 `FIN`。
- 本端内核也回了 `ACK`。
- 但应用层还没有真正调用 `close()` 把连接彻底关闭。

所以如果线上长期积累大量 `CLOSE_WAIT`，经验上首先要怀疑：

- 应用代码忘了关闭连接。
- 某些异常分支、超时分支没有释放 socket。
- 上层框架或连接池管理有 bug。

一句话概括：

- `TIME_WAIT` 更多是协议正常行为导致的资源现象。
- `CLOSE_WAIT` 很多时候是应用没有把该收的尾收干净。

## 6. `TIME_WAIT` 在服务端和客户端谁更常见

谁主动关闭，谁更容易进入 `TIME_WAIT`。

所以不能机械地说“总是客户端有 `TIME_WAIT`”或者“总是服务端有 `TIME_WAIT`”。

例如：

- 浏览器访问 Web 服务，常见情况是客户端主动断开，因此客户端更容易出现 `TIME_WAIT`。
- 如果服务端有超时踢连接、主动关闭短连接、代理层主动回收连接等行为，服务端也可能出现大量 `TIME_WAIT`。

面试时讲到这里，说明你理解的是“主动关闭者承担 `TIME_WAIT`”，而不是死记某个固定角色。

## 7. 能不能直接优化掉 `TIME_WAIT`

高级工程师不应该把答案说成“调一下 sysctl 就行”。更稳妥的回答是：

1. 先确认 `TIME_WAIT` 是否真的构成瓶颈，而不是正常现象。
2. 再看是否存在短连接过多、连接复用不足、代理配置不合理的问题。
3. 最后才讨论内核参数调优，而且要明确风险边界。

很多线上事故都来自“没理解协议语义就直接压参数”。网络调优不是参数竞赛，前提是先分清根因。

## 8. 优雅关闭服务和这部分知识有什么关系

优雅关闭不是简单 `kill -9`。

一个成熟服务停机通常要做：

1. 从注册中心或负载均衡摘流。
2. 停止接收新连接。
3. 让现有连接处理完在途请求。
4. 超过截止时间后，再强制清理剩余连接。

如果你不理解 TCP 断连状态和半关闭语义，就很容易把“优雅关闭”做成“随机断流”。

## 9. 面试里更像高级工程师的表达方式

如果面试官问“为什么会有 `TIME_WAIT`”，可以按这个结构答：

1. 先说它通常出现在主动关闭方。
2. 再说两个目的：保证最后 ACK 可重发、隔离旧报文。
3. 然后结合线上场景，说明大量短连接时为什么会堆积。
4. 最后给出治理思路：连接复用优先，参数调优谨慎。

如果还能顺手对比 `CLOSE_WAIT`，基本就已经不是背诵水平，而是可用于排障的工程理解。

# WebSocket、SSE、流式传输、背压与代理兼容性

定位：这组知识直接服务于实时推送、AI 流式输出、日志订阅、事件通知和长连接网关设计。很多系统“功能能跑”，但一上代理层、弱网、长连接高并发或慢消费者场景就暴露问题，本质上往往是没有把协议语义、背压和中间层行为想清楚。

## 1. 先分清三类东西：WebSocket、SSE、普通 HTTP 流式传输

很多面试回答一上来就把它们混成“都是长连接推送”，这不够准确。

### 1.1 WebSocket

- 目标是建立一个全双工、长生命周期的双向通信通道。
- 客户端和服务端都可以主动发消息。
- 常见于聊天、协作编辑、实时控制、双向事件通道。

### 1.2 SSE

- Server-Sent Events，本质上是服务端到客户端的单向事件流。
- 仍然建立在 HTTP 语义上，浏览器端通常用 `EventSource` 消费。
- 适合通知流、日志流、AI 文本流式输出这类“服务端持续推、客户端只接收”的场景。

### 1.3 普通 HTTP 流式传输

- 不一定遵循 SSE 事件格式，也不一定升级为 WebSocket。
- 可以是 chunked response、HTTP/2 stream、逐段输出文本或二进制内容。
- 常用于下载、增量返回、模型 token 流。

一句话区分：

- WebSocket 强在双向实时交互。
- SSE 强在简单的服务端单向推送。
- 普通 HTTP 流式传输是更泛化的“边生成边返回”。

## 2. WebSocket 的本质是什么

WebSocket 不是“另一个完全无关的 TCP 协议”，它通常从 HTTP 握手开始：

1. 客户端先发起一个带 `Upgrade: websocket` 的 HTTP 请求。
2. 服务端同意升级后，双方切换到 WebSocket 帧协议。
3. 后续通信不再是普通 HTTP 请求/响应模式，而是持久双向消息通道。

这也是为什么代理兼容性会成为高频问题：

- 某些代理对普通 HTTP 很熟，但对 upgrade、长连接、空闲超时、帧转发支持并不完善。

## 3. SSE 的本质是什么

SSE 并不会把协议升级成别的东西，它仍然是 HTTP 响应，只是：

- 响应头通常是 `Content-Type: text/event-stream`
- 服务端不一次性返回完整 body，而是持续写入事件块
- 客户端按事件流实时消费

所以 SSE 相比 WebSocket 的优势是：

- 更简单
- 更贴近 HTTP 生态
- 对很多代理和浏览器更友好

但它也有清晰边界：

- 主要是单向服务端推送
- 不适合高频双向互动

## 4. SSE 和 WebSocket 怎么选

这题不能只答“WebSocket 更强大”。更工程化的判断是：

### 4.1 优先选 SSE 的场景

- 服务端单向推送
- 浏览器消费
- 实时日志、通知、AI token 流
- 希望尽量复用 HTTP 基础设施

### 4.2 优先选 WebSocket 的场景

- 双向实时交互
- 客户端也要频繁发消息
- 低延迟双向事件通道
- 聊天、协作编辑、游戏、控制信令

### 4.3 普通 HTTP streaming 适合的场景

- 只是想逐段返回结果，不一定需要标准化事件格式
- 非浏览器客户端也很多
- 与现有 HTTP/gRPC 流接口更容易整合

## 5. 什么叫流式传输里的背压

背压的本质问题是：

- 生产数据的速度大于消费数据的速度。

在流式场景里，这会表现成：

- 服务端持续生成消息
- 客户端读得慢
- 中间代理或内核发送缓冲区逐渐堆积
- 最终导致内存增长、连接阻塞、整体吞吐下降甚至雪崩

所以背压不是“高级框架才关心的抽象概念”，而是任何长连接和流式传输系统都必须面对的资源控制问题。

## 6. 背压在系统里通常出现在哪几层

### 6.1 应用层

- 上游生成事件太快
- 下游业务逻辑处理慢

### 6.2 进程内缓冲

- channel、队列、消息缓存不断积压

### 6.3 socket 发送缓冲区

- 对端读得慢，导致写操作阻塞或累计 pending bytes

### 6.4 代理层缓冲

- 中间代理为了转发或缓存，先把数据攒起来
- 这可能破坏“实时流式”效果，甚至放大内存占用

所以真正的背压治理必须是端到端的，不能只看应用代码。

## 7. 慢消费者为什么危险

慢消费者是流式系统里的经典问题：

- 单个客户端读得很慢
- 服务端为了照顾它持续缓存待发送数据
- 结果一个慢连接拖累整个进程或整个广播通道

典型后果包括：

- 单连接内存暴涨
- 广播延迟上升
- 事件循环被写阻塞拖慢
- 热点连接拖垮整体服务

高级工程师应该知道几个常见处理方向：

- 限制单连接缓冲上限
- 达到阈值就断开慢连接
- 做采样、丢弃旧消息或合并更新
- 通过消息队列把生产和发送解耦

## 8. 代理兼容性为什么是流式系统的高风险点

因为代理经常改变你以为的传输语义。

常见问题包括：

- 代理层默认 buffering，导致 SSE 或 chunked 数据攒一大块才发出去
- 空闲超时太短，把看似正常的长连接断掉
- 不支持或没正确配置 WebSocket upgrade
- 连接数和 fd 上限不足
- 代理层重试、重连、健康检查与长连接语义冲突

所以很多“本地调试没问题，线上流不出来”的问题，根因都在代理或网关配置，而不是业务代码。

## 9. SSE 常见代理问题有哪些

SSE 常见问题非常典型：

- 代理对响应做 buffer，客户端迟迟收不到实时事件
- gzip/压缩策略不当，导致流式输出粒度被破坏
- idle timeout 太短，长时间无事件时连接被断开
- 反向代理把连接当普通短请求处理

所以做 SSE 时经常要显式配置：

- 关闭或限制 proxy buffering
- 合理设置超时
- 定期发送心跳事件，防止中间层误判为空闲死连接

## 10. WebSocket 常见代理问题有哪些

WebSocket 最典型的问题是：

- upgrade 头没有透传
- 代理不支持或未正确启用长连接升级
- 空闲超时导致连接莫名断开
- 负载均衡和会话粘性策略不匹配，导致连接重建或状态错乱

如果系统还依赖单机内存态会话，WebSocket 在扩缩容和负载均衡上的复杂度会明显上升。

## 11. 流式 AI 输出为什么常用 SSE

很多 AI 应用选择 SSE，不是因为它最强，而是因为它足够简单且契合场景：

- 服务端单向持续吐 token
- 浏览器接入方便
- 比 WebSocket 更容易复用现有 HTTP 代理与鉴权体系

但一旦你要做：

- 客户端中途打断
- 双向控制信令
- 大规模双向协同

WebSocket 或更复杂的实时协议设计就会更合适。

## 12. 怎么设计流式传输的背压策略

一个比较成熟的设计通常会包含：

1. 对单连接待发送缓冲做上限控制。
2. 区分关键消息和可丢弃消息。
3. 慢消费者达到阈值后限速、降采样或断开。
4. 将数据生产速率与网络发送速率解耦。
5. 在代理、应用、客户端三层统一考虑超时与缓冲策略。

真正的目标不是“永不丢消息”，而是“系统整体稳定、关键语义正确”。

## 13. 面试里更像高级工程师的答法

如果被问“WebSocket 和 SSE 区别”，建议不要停在“一个双向一个单向”。更完整的答法是：

1. 先说 WebSocket 是升级后双向长连接，SSE 是基于 HTTP 的单向事件流。
2. 再说它们各自适用场景。
3. 补充普通 HTTP streaming 作为第三类方案。
4. 解释流式场景的核心风险是背压和慢消费者。
5. 最后强调代理层 buffering、idle timeout、upgrade 支持等兼容性问题。

这样回答已经从协议差异走到了线上工程问题。

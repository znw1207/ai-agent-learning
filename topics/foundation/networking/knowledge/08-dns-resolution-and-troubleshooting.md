# DNS 解析全过程与常见故障

定位：DNS 不是“背一个域名转 IP 的概念题”就结束了。高级后端工程师要把它看成请求真正进入服务前的第一跳控制面，很多线上“偶发超时”“切流不生效”“只有部分机器失败”的问题，本质都和 DNS 有关。

## 1. 一次域名解析大致经历了什么

一个应用要访问 `api.example.com`，通常不会自己直接去找根域名服务器，而是先走本地解析链路：

1. 先查本地缓存，例如进程内缓存、语言运行时缓存、操作系统缓存。
2. 若未命中，再查本地配置，例如 `/etc/hosts`。
3. 再把请求交给本机配置的递归解析器，通常是公司内 DNS、路由器 DNS、云厂商 DNS 或公共 DNS。
4. 递归解析器若没有缓存，才会代表客户端继续向外做迭代查询。
5. 最终拿到目标记录后返回给客户端，并按 TTL 缓存。

所以从应用视角看，往往只发起“一次 DNS 查询”，但递归解析器背后可能做了多跳查询。

## 2. 递归查询和迭代查询的区别

### 2.1 递归查询

- 客户端把“帮我查到最终结果”这个责任交给递归解析器。
- 递归解析器负责自己继续往下问，直到拿到最终答案或失败。

### 2.2 迭代查询

- 某个 DNS 服务器不直接给最终答案，而是告诉你“下一步去问谁”。
- 例如根服务器告诉你去问 `.com` 顶级域服务器，顶级域服务器再告诉你权威服务器在哪里。

工程上客户端通常面对的是递归查询，递归解析器对外执行的是迭代查询。这两个层次不要混。

## 3. 根、顶级域、权威 DNS 各自负责什么

- 根 DNS：知道顶级域去哪找。
- 顶级域 DNS：知道某个具体域名的权威 DNS 在哪。
- 权威 DNS：掌握最终记录，例如 `A`、`AAAA`、`CNAME`、`MX` 等。

很多面试只问到这里，但高级工程师应该继续补一句：

- 真正影响业务稳定性的，往往不是根和顶级域，而是递归解析器缓存策略、权威记录配置、TTL 和客户端缓存行为。

## 4. 常见记录类型应该掌握到什么程度

至少要熟悉：

- `A`：域名到 IPv4 地址。
- `AAAA`：域名到 IPv6 地址。
- `CNAME`：别名记录，指向另一个域名。
- `MX`：邮件服务器记录。
- `NS`：指定某个域的权威 DNS。
- `TXT`：常用于校验、SPF、DKIM、配置透传。

后端面试里最常用的是 `A`、`AAAA`、`CNAME` 和 `NS`。

## 5. TTL 为什么重要

TTL 决定了解析结果能被缓存多久。它直接影响两类工程问题：

### 5.1 TTL 太长

- 切流慢。
- 故障实例下线后，旧 IP 还会被继续命中。
- 灰度、迁移、容灾切换变慢。

### 5.2 TTL 太短

- 查询量明显上升。
- 递归 DNS 压力变大。
- 应用更频繁依赖解析链路，放大 DNS 波动影响。

所以 TTL 不是越小越灵活，也不是越大越稳定，而是一个切换速度与解析负担之间的工程权衡。

## 6. 为什么“改了 DNS 但流量没立刻切走”

这是非常高频的线上问题。根因通常不是“DNS 没生效”，而是缓存层次太多：

- 客户端库缓存。
- JVM / Go / Nginx / Envoy 等运行时缓存。
- OS 缓存。
- 本地 DNS 缓存。
- 递归解析器缓存。
- 中间代理缓存。

所以你即使改了权威记录，只要某一层还没过 TTL，流量就可能继续打到旧地址。

高级工程师回答这类题时，不能只说“等 TTL 过期”，而要意识到多层缓存叠加会让生效时间比你看到的权威配置更复杂。

## 7. 为什么有时只有部分机器解析失败

这种问题很常见，也最能区分工程经验。常见原因包括：

- 某个机房或某台机器使用了异常的递归 DNS。
- `/etc/hosts` 或本地配置被污染。
- IPv4/IPv6 偏好不同，导致解析路径不一致。
- 运行时 DNS 缓存没有刷新。
- 某些节点走了不同的网络出口或不同的 Service Mesh。

所以看到“只有部分机器失败”，第一反应不该是“DNS 全局挂了”，而是“是不是解析路径、缓存层或网络出口不一致”。

## 8. DNS 慢通常慢在哪

DNS 慢不一定是权威 DNS 慢。常见耗时点包括：

- 本地递归 DNS 负载高。
- 缓存未命中，触发外部多跳查询。
- 上游权威 DNS 响应慢。
- 网络链路抖动。
- 查询超时后 fallback 到其他 nameserver。
- AAAA / A 双栈尝试和超时策略不当。

很多应用日志里只看到“连接超时”，但真正慢的是前面的 DNS。没有阶段化观测时，这类问题特别容易误判。

## 9. DNS 和服务发现有什么关系

在很多现代系统里，DNS 不只是公网域名解析，还承担服务发现职责，例如：

- K8s Service 通过集群 DNS 暴露服务名。
- 内网服务通过 DNS 轮询或 SRV 风格记录做发现。
- 网关、Sidecar、代理可能周期性重解析目标地址。

所以 DNS 的稳定性会直接影响 RPC、HTTP 调用、负载均衡和故障切换。

## 10. 高级工程师如何排查 DNS 问题

一个靠谱的排查顺序通常是：

1. 先确认失败是不是解析阶段而不是连接阶段。
2. 对比应用侧、OS 侧、`dig/nslookup` 看到的结果是否一致。
3. 核对 `/etc/hosts`、resolv 配置、本地 DNS 和容器网络配置。
4. 查 TTL、权威记录、递归缓存和是否存在多层缓存未刷新。
5. 对比不同机器、不同机房、不同网络出口的解析结果。

这类题真正考的不是术语，而是你能不能把“域名失败”拆成具体链路步骤来定位。

## 11. 面试里更像高级工程师的答法

如果被问“DNS 解析过程是怎样的”，建议这样答：

1. 先说客户端一般先查本地缓存和 hosts。
2. 再说请求通常交给递归解析器。
3. 解释递归解析器如何通过根、TLD、权威 DNS 做迭代查询。
4. 再补 TTL 和多层缓存为什么会影响切流和故障恢复。
5. 最后带一句常见故障：部分机器解析失败、TTL 未到、权威配置改了但应用没刷新缓存。

这样才像真实工程答法，而不是纯教科书。

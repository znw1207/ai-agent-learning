# TCP 重传、流量控制与拥塞控制

定位：这是把“TCP 是可靠的”这句话拆开讲清楚的关键部分。后端工程师如果只会背三次握手四次挥手，但说不清 TCP 为什么可靠，面试深挖通常就会断层。

## 1. TCP 为什么被称为可靠传输

TCP 的可靠，不是“永不丢包”，而是：

- 即使底层网络可能丢包、乱序、重复，TCP 仍尽量向上层提供有序、可靠的字节流语义。

它主要依赖几类机制：

- 序列号
- 确认应答（ACK）
- 重传机制
- 滑动窗口
- 流量控制
- 拥塞控制

高级工程师回答 TCP 可靠性时，不能只说“有 ACK 所以可靠”，那太浅。

## 2. 重传机制在解决什么问题

网络里报文可能丢失，也可能 ACK 丢失。如果发送方一直等而不重发，数据就永远卡住了。

所以 TCP 需要重传机制来保证：

- 已发送但未被确认的数据，最终有机会再次送达。

最基本的方式是超时重传：

- 发送方发出报文后启动计时器。
- 在合理时间内没收到确认，就认为可能丢失并重发。

但只靠超时重传会偏慢，所以 TCP 还会结合其他信号更快判断丢包。

## 3. 什么是快速重传

如果接收方发现报文乱序，通常会重复确认当前连续收到的最后一个位置。

发送方如果连续收到多个重复 ACK，就会推测：

- 某个中间报文大概率丢了。

于是它可以不等重传超时，直接提前重发丢失段。这就是快速重传。

它的价值在于：

- 让 TCP 不必完全依赖超时，能更快从单个丢包中恢复。

## 4. 滑动窗口在解决什么问题

如果发送方每发一个报文都必须停下来等 ACK，再发下一个，吞吐会很差。

滑动窗口的意义是：

- 允许发送方在未收到全部 ACK 前，连续发送一段范围内的多个字节或报文。

这样链路可以被“灌满”，显著提高吞吐。窗口往前滑动，代表已经被确认的数据可以移出，新的数据可以继续发送。

所以窗口机制本质上是在可靠性和吞吐之间做平衡，而不是单纯为了“限流”。

## 5. 流量控制（Flow Control）是什么

流量控制关注的是：

- 接收方来不来得及收。

即使网络本身没拥塞，如果接收方应用读取太慢、接收缓冲区快满了，发送方也不能无脑继续推数据。

所以接收方会通过接收窗口告诉发送方：

- 我现在还能接收多少数据。

发送方据此控制发送速度。它解决的是端到端两端处理能力不匹配的问题。

## 6. 拥塞控制（Congestion Control）是什么

拥塞控制关注的是：

- 网络本身是否扛得住。

这和流量控制不同。流量控制看的是接收端缓冲能力，拥塞控制看的是路径中间网络设备、链路和整个网络状态。

如果网络已经拥塞，继续猛发只会带来更多丢包、重传和更差的整体吞吐。因此 TCP 会根据丢包、时延、ACK 等信号动态调整拥塞窗口，控制注入网络的数据量。

一句话区分：

- 流量控制防止把接收方压垮。
- 拥塞控制防止把网络压垮。

## 7. 为什么慢启动不是“慢慢地一点点发”

慢启动名字容易误导。它并不是线性慢速增长，而是初期按较快节奏扩大可发送量，以尽快探测可用带宽。

其核心思想是：

- 刚开始别一口气打满网络。
- 先逐步探测。
- 一旦发现拥塞信号，再收缩发送窗口。

面试里如果只说“慢启动就是开始发得慢一点”，说明没有真正理解它是拥塞控制探测机制的一部分。

## 8. 为什么丢包不一定只代表网络差

很多人把丢包和“链路坏了”直接画等号。实际工程里，丢包可能来自：

- 真正的链路质量问题。
- 交换机或路由器缓存溢出。
- 接收端处理不过来。
- 队列过长导致主动丢弃。
- 网络设备策略限制。

所以看到重传增多、吞吐下降时，不能机械判断为“网不行”，必须结合 RTT、队列、主机资源和应用负载综合判断。

## 9. 面试里如何把流量控制和拥塞控制讲清楚

比较好的结构是：

1. 先说 TCP 可靠性依赖 ACK、重传和窗口机制。
2. 再区分两个控制目标。
3. 流量控制面向接收端能力。
4. 拥塞控制面向网络整体状态。
5. 最后补充快速重传、慢启动等恢复与探测机制。

这样回答比单独背几个术语更有系统性。

## 10. 这些知识怎么映射到后端工程实践

- 为什么高延迟链路吞吐差，不只是“带宽不够”，还和窗口利用率有关。
- 为什么跨地域调用、弱网环境、代理转发会引入额外重传和时延抖动。
- 为什么请求超时和重试策略不能脱离网络现实来设计。
- 为什么大文件传输、流式传输和普通 RPC 在链路表现上差异很大。

理解这部分后，你对“TCP 稳定性”的认识才算从概念走向工程。

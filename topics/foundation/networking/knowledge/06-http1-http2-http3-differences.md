# HTTP/1.1、HTTP/2、HTTP/3 的核心差异

定位：这部分知识不是背协议版本号，而是理解现代服务链路里延迟、连接复用、队头阻塞和传输层选择对系统表现的影响。

## 1. 先抓住主线，不要陷入细节碎片

如果面试官问 HTTP/1.1、HTTP/2、HTTP/3 的差异，最好的回答方式不是逐条背特性，而是沿着这条主线讲：

1. 它们如何复用连接。
2. 它们如何组织请求与响应。
3. 它们如何面对队头阻塞。
4. 它们底层跑在什么传输层之上。
5. 这些差异对真实系统延迟和吞吐有什么影响。

## 2. HTTP/1.1 的核心特点

### 2.1 文本协议，语义直接

- 请求行、Header、Body 都是文本形式表达。
- 调试和人类理解相对直观。

### 2.2 支持长连接，但复用能力有限

- HTTP/1.1 默认可复用 TCP 连接，也就是 Keep-Alive 思想。
- 但同一条连接上，请求响应通常仍偏顺序化。
- 早期虽然有 pipeline 概念，但实际工程里使用很少。

### 2.3 应用层队头阻塞明显

- 一条连接上前一个请求没处理完，后面的请求就容易被卡住。
- 所以浏览器和客户端通常会开多个连接并发请求。

### 2.4 头部冗余较大

- 每个请求都会重复发送较多 Header。
- 对小请求、高频调用场景不够友好。

一句话总结：HTTP/1.1 能工作、也够稳定，但在高并发、多资源、低延迟场景下，连接复用和并发效率有限。

## 3. HTTP/2 的核心改进

### 3.1 二进制分帧

- 不再直接按纯文本报文组织，而是拆成更底层的二进制 frame。
- 协议机器更容易高效解析和复用。

### 3.2 多路复用

- 多个请求/响应可以并发复用同一条 TCP 连接。
- 每个请求被标识为不同 stream。
- 不再需要像 HTTP/1.1 那样频繁多开连接来提升并发。

### 3.3 Header 压缩

- 减少重复头部带来的带宽浪费。
- 对高频请求、微服务场景、网关调用都有实际价值。

### 3.4 Server Push 曾被寄予厚望，但现实价值有限

- 协议支持服务端主动推送资源。
- 但实际工程效果一般，很多场景收益不稳定，复杂度不低。

### 3.5 仍然基于 TCP

- HTTP/2 解决了 HTTP/1.1 的应用层队头阻塞。
- 但它仍然跑在单条 TCP 连接之上。
- 如果 TCP 层出现丢包，整条连接上的多个 stream 仍会一起受影响。

这就是为什么很多人说 HTTP/2 解决了一部分队头阻塞，但没有彻底解决传输层层面的队头阻塞。

## 4. HTTP/3 的核心变化

### 4.1 从 TCP 切换到 QUIC/UDP

- HTTP/3 不再建立在 TCP 之上，而是建立在 QUIC 之上。
- QUIC 运行在 UDP 上，但自己实现了可靠传输、流控、拥塞控制和安全能力。

### 4.2 把传输层和安全层现代化整合

- QUIC 把很多原来 TCP + TLS 分开的能力整合在一起设计。
- 握手与连接迁移等体验更适合现代网络环境，尤其是移动网络。

### 4.3 更好地缓解传输层队头阻塞

- HTTP/2 虽然有 stream，但底层还是一条 TCP 字节流。
- 一旦 TCP 某个报文丢失，后续字节交付会被整体拖住。
- QUIC 则按流管理交付，某个流受丢包影响，不必把所有流都一起卡死到同样程度。

### 4.4 更适合弱网和连接迁移场景

- 移动端网络切换、NAT 变化、链路抖动下，QUIC 的体验通常更灵活。
- 这也是 HTTP/3 在现代互联网场景的重要价值之一。

## 5. 队头阻塞到底怎么区分

这个点面试特别容易问乱。你可以这样分：

### 5.1 HTTP/1.1 的应用层队头阻塞

- 同一连接中前面请求没处理完，后面请求就难以前进。

### 5.2 HTTP/2 解决了应用层队头阻塞

- 多个 stream 可以并发复用一条连接。

### 5.3 HTTP/2 仍受 TCP 层队头阻塞影响

- TCP 丢一个关键报文，后续字节可能都要等重传。
- 因为多个 stream 共用这条 TCP 字节流，所以会一起感受到影响。

### 5.4 HTTP/3/QUIC 进一步缓解传输层队头阻塞

- 因为流管理和重传组织方式不同，单个流的问题不必完全拖住其他流。

如果你能把这四层关系讲清楚，说明你不是只记了“HTTP/3 更快”这种口号。

## 6. 为什么 HTTP/2 不一定绝对优于 HTTP/1.1

协议升级不是单向神话。HTTP/2 虽然通常更先进，但也不是所有场景都天然收益更大。

比如：

- 链路极其简单、请求模式非常单一时，收益可能没有想象中大。
- 中间代理、网关、TLS 终止层、老旧基础设施可能带来额外兼容和调优成本。
- 某些极端丢包场景下，单 TCP 连接承载大量 stream 也会放大单连接问题。

高级工程师回答时，应该体现“协议能力”和“系统整体收益”不是完全等价。

## 7. 为什么 HTTP/3 也不是一键无脑切换

HTTP/3 引入 QUIC 后，收益明显，但成本也真实存在：

- 监控与排障手段会变化。
- 中间网络设备和代理支持情况要评估。
- UDP 流量策略、限流和防火墙规则需要重新审视。
- 团队对 QUIC 的观测与调优经验可能不足。

所以在生产里，HTTP/3 往往是“逐步接入、灰度验证、按场景收益推进”，而不是看到新版本就全面切换。

## 8. 面试里更像高级工程师的答法

如果被问“HTTP/1.1、2、3 最大差异是什么”，建议按这个结构回答：

1. HTTP/1.1：长连接但并发复用能力有限，应用层队头阻塞明显。
2. HTTP/2：二进制分帧 + stream 多路复用 + Header 压缩，改善应用层并发效率。
3. HTTP/2 仍跑在 TCP 上，所以仍受 TCP 层队头阻塞影响。
4. HTTP/3 基于 QUIC/UDP，进一步缓解传输层队头阻塞，更适合现代弱网和移动环境。
5. 最后补一句：协议升级要结合链路质量、网关设施、观测能力和业务收益评估。

这样回答比背一堆特性列表更有工程感。

## 9. 这部分知识映射到哪些工程场景

- 网关和 API 层为什么关注连接复用和握手延迟。
- 浏览器端资源加载为什么协议升级会明显影响首屏表现。
- 微服务网关、边缘代理、CDN 为什么会关注 H2/H3 支持。
- 弱网、跨地域、移动场景为什么更关心 QUIC 的实际收益。

# TCP 建连与服务端处理流程

定位：这是后端工程师网络基础里的高频主干知识，直接服务于高并发服务、超时治理、连接异常排障、限流与负载均衡设计。

## 1. 先建立一个正确心智模型

一次客户端访问服务端，不是“业务代码先收到请求，再去建立连接”，而是：

1. 服务端进程先通过 `socket()`、`bind()`、`listen()` 把监听端口交给内核。
2. 客户端发起 TCP 三次握手。
3. 握手主要由内核 TCP 协议栈处理，不是业务线程手写处理 SYN/ACK。
4. 握手完成后，连接进入可被应用层 `accept()` 取走的状态。
5. 应用层拿到已连接 socket 后，才开始 `read`/`recv` 读取 HTTP、RPC 或自定义协议数据。

所以，监听 socket 负责“接收新连接”，已连接 socket 负责“收发具体业务数据”。这是很多初中级工程师最容易混淆的点。

## 2. 服务端在建连前做了什么

### 2.1 `socket()`

- 向内核申请一个 socket 对象。
- 例如 `AF_INET` + `SOCK_STREAM` 代表 IPv4 TCP。
- 此时它还没有端口，也不能接收外部连接。

### 2.2 `bind()`

- 把本地 IP 和端口绑定到这个 socket。
- 典型形式是监听 `0.0.0.0:8080`，表示在本机所有网卡的 8080 端口接收连接。
- 如果端口已被占用，通常会在这里失败。

### 2.3 `listen()`

- 把普通 socket 变成监听 socket。
- 告诉内核：这个端口现在开始用于被动接收连接。
- 同时给内核一个 backlog 提示，用于连接排队控制。

到这里，服务端应用层的准备动作就完成了。后面的 TCP 握手主要由内核完成。

## 3. TCP 三次握手期间，谁在做事

### 3.1 客户端发 SYN

- 客户端发起建连，请求与服务端某个 IP:port 建立 TCP 连接。

### 3.2 服务端内核回 SYN+ACK

- 服务端内核发现该目标端口处于监听状态。
- 内核为该连接创建半连接状态，回复 `SYN+ACK`。
- 这里一般还没有把连接交给用户态业务代码。

### 3.3 客户端回 ACK

- 客户端再回一个 ACK。
- 至此三次握手完成，连接变成 established。
- 此时连接才具备被 `accept()` 返回给应用层的条件。

核心点：三次握手是内核网络栈在主导，业务代码只是提前把监听端口准备好，然后通过 `accept()` 消费结果。

## 4. 半连接队列与全连接队列

面试里常问“服务端在握手过程中会不会排队”，标准答案通常要讲两个阶段。

### 4.1 半连接队列

- 用来存放收到 SYN、但还没完成三次握手的连接。
- 常被称为 SYN queue 或 request queue。
- 如果这里被打满，可能触发丢包、SYN cookies、握手失败等现象。

### 4.2 全连接队列

- 用来存放已经完成三次握手、但应用层还没来得及 `accept()` 的连接。
- 常被称为 accept queue。
- 如果应用层处理不及时，全连接队列可能积压，导致新的已完成握手连接无法及时入队。

注意：具体实现细节和命名依赖内核版本，但工程上你应该掌握“未完成握手”和“已完成握手未被 accept”是两个不同阶段。

## 5. `listen(backlog)` 到底控制什么

这题不能回答成一句“就是队列长度”。更准确的说法是：

- `backlog` 是应用层传给内核的排队上限提示，不是绝对值。
- Linux 实际会受系统参数共同影响，例如 `somaxconn`、SYN 相关参数等。
- 工程上通常把它理解为“已完成握手但尚未被 accept 的连接队列上限”的核心控制参数。
- 但它与未完成握手阶段的表现也有关联，因为整个监听 socket 的排队能力最终是系统协同结果，不是单一数字决定。

高级一点的回答应该再补一句：

- 当 `accept()` 不及时、全连接队列堆积时，看起来像“服务还能握手，但业务处理不过来”。
- 当半连接压力大时，更像 SYN 洪泛、握手超时、建连失败率上升的问题。

## 6. `accept()` 做的是什么

- `accept()` 不是“开始建立连接”，而是“从内核取走一个已经准备好的连接”。
- 返回的是一个新的已连接 socket fd。
- 原监听 socket 仍然继续监听后续新连接。

因此服务端通常会同时持有：

- 一个监听 socket：只负责接新连接。
- 多个已连接 socket：每个代表一个客户端会话。

这是服务端并发模型的基础，不管是多进程、线程池、`select/poll/epoll`，还是 Go runtime netpoll，本质都绕不开这一点。

## 7. 阻塞与非阻塞的区别

### 7.1 阻塞 `accept()`

- 如果当前没有已完成连接，线程会睡眠等待。
- 传统简单服务模型很常见。

### 7.2 非阻塞 `accept()`

- 没有连接时直接返回 `EAGAIN`/`EWOULDBLOCK`。
- 常与 `epoll` 搭配，先由事件通知“有连接可取”，再去 `accept()`。

面试时要体现工程感：

- 高并发服务器不会靠大量线程傻等在阻塞 `accept()` 上。
- 更常见是事件驱动 + 非阻塞 fd + 多路复用。

## 8. 从 TCP 建连到 HTTP 请求的边界

很多人会把“连接建立”和“收到 HTTP 请求”混成一件事。实际上是两段：

1. TCP 建连：解决的是“双方有没有一条可靠字节流通道”。
2. HTTP 解析：解决的是“这条字节流里的协议内容是什么”。

所以 `accept()` 返回成功，只说明链路建立了，不代表你已经读到了完整 HTTP 请求头，更不代表业务请求已经处理完成。

## 9. 面试里怎么答得像高级工程师

如果题目是“说一下客户端连到服务端的过程”，一个更完整的回答结构应该是：

1. 先说服务端预先执行 `socket/bind/listen`。
2. 再说 TCP 三次握手主要由内核完成。
3. 解释半连接和全连接两个阶段。
4. 说明 `accept()` 只是取走已完成连接，并返回新的连接 fd。
5. 最后补充阻塞/非阻塞、`epoll`、backlog、队列打满后的工程后果。

这样回答，才从“背协议流程”提升到了“理解服务端运行机制”。

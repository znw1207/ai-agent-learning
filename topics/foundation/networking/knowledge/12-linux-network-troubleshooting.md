# Linux 网络排障主线

定位：高级工程师和中级工程师最容易拉开差距的地方，不是能不能背协议，而是线上出问题时能不能快速判断是 DNS、建连、队列、连接池、代理、内核参数还是应用自身导致。

## 1. 排障先按阶段拆，不要上来就猜

一次请求失败，通常可以拆成这些阶段：

1. DNS 是否正常。
2. TCP 建连是否正常。
3. TLS 握手是否正常。
4. 请求是否成功发出。
5. 服务端是否及时处理。
6. 响应是否及时返回。
7. 客户端是否正确读取。

如果不先把问题按阶段拆开，排障很容易退化成“感觉像网络问题”。

## 2. 常见现象和第一判断方向

### 2.1 CPU 不高，但 RT 很高

优先考虑：

- 下游慢
- 线程/连接在等待
- 锁竞争
- 连接池打满
- DNS 或建连耗时被算进总 RT

### 2.2 连接数暴涨，但 QPS 没涨

优先考虑：

- 长连接回收异常
- 下游响应变慢导致连接占用时间拉长
- 连接泄漏
- 代理层或客户端复用失效

### 2.3 大量 `TIME_WAIT`

优先考虑：

- 短连接过多
- 主动关闭过多
- 某层 keepalive/连接池复用异常

### 2.4 大量 `CLOSE_WAIT`

优先考虑：

- 应用层没正确关闭连接
- 异常路径泄漏 fd

## 3. 为什么“网络问题”很多时候其实是应用问题

线上很多看似网络故障的现象，实际上是：

- 线程池耗尽
- GC 抖动
- 锁竞争
- 下游慢 SQL
- 连接未归还
- 超时配置错误

它们最终在现象上表现为超时、重传增多、连接堆积，但根因未必在网卡或链路本身。

所以高级工程师排障时，一定会把主机资源、应用线程、连接池和网络观测一起看。

## 4. 线上最该关注哪些信号

建议最少建立这些观测：

- DNS 耗时
- 建连耗时
- TLS 握手耗时
- 请求发送耗时
- 首包时间
- 总 RT
- 重试次数
- 连接池等待时间
- 活跃连接/空闲连接/等待队列
- TCP 状态分布
- 丢包、重传、RTT 抖动

没有分阶段观测，排障基本只能靠猜。

## 5. 常见排障工具要理解到什么程度

高级工程师不一定每天手敲所有命令，但要知道这些工具能回答什么问题：

- `ss` / `netstat`：连接状态、端口、队列。
- `lsof`：谁持有 fd。
- `tcpdump`：抓包看握手、重传、RST、时序。
- `dig` / `nslookup`：查 DNS 结果和权威链路。
- `top` / `pidstat` / `vmstat`：CPU、上下文切换、负载。
- `sar` / `iostat`：系统级资源变化。
- 应用 trace / profile / logs：定位是网络慢还是业务慢。

工具不是为了炫技，而是为了把“怀疑”变成“证据”。

## 6. `somaxconn`、fd 上限、socket buffer 为什么常被问

因为它们经常和接入能力直接相关：

- `somaxconn` 影响监听队列上限协同表现。
- fd 上限影响进程能持有多少连接。
- socket buffer 影响吞吐、突发流量承接和时延表现。

但高级工程师不应把答案说成“参数越大越好”。参数调整必须基于现象、指标和容量模型，不然只是把问题从一个位置移到另一个位置。

## 7. 遇到 RT 抖动时，怎么避免误判

一个常见错误是：

- 看到 RT 高，就认为业务代码慢。

但真正成熟的判断会先拆：

- 是 DNS 慢？
- 是建连慢？
- 是连接池等连接慢？
- 是请求发出后对端处理慢？
- 是响应返回慢？

把 RT 分段后，问题范围会立刻小很多。

## 8. 面试里更像高级工程师的答法

如果被问“线上服务 RT 抖高你怎么查”，建议这样答：

1. 先按 DNS、建连、TLS、请求处理、响应返回几个阶段拆。
2. 同时看应用层和系统层指标，不要只盯一边。
3. 对比 RT、连接数、连接池等待、TCP 状态、重试次数、下游错误率。
4. 判断是链路问题、资源问题、连接管理问题还是应用本身变慢。
5. 最后再决定是否看内核参数、抓包或做更细观测。

这类题考的是方法论，不是命令记忆。

# 网络面试题 03：IO 多路复用与 epoll

## 题目 1：为什么高并发服务器通常不用“一连接一线程”的模型？

高级工程师回答：

因为这个模型的主要问题不在于“不能工作”，而在于扩展成本太高。连接一多，就会产生大量线程，而大量线程大部分时间其实只是阻塞在 `read`、`write` 或 `accept` 上等待 IO，就会带来高内存占用、频繁上下文切换和调度开销。也就是说，系统把很多资源花在“等待”上，而不是花在真正处理请求上。

高并发服务器更合理的目标是让少量线程管理大量连接，只在 fd 真正可读、可写或有新连接到达时才被唤醒工作。这样才能把等待成本从“线程级”压缩到“事件级”。这也是为什么高性能网络服务通常会走事件驱动和 IO 多路复用路线。

## 题目 2：阻塞 IO、非阻塞 IO 和 IO 多路复用分别是什么？

高级工程师回答：

阻塞 IO 的特点是：如果 fd 当前没有数据，`read()` 会把线程挂起，直到数据到达。非阻塞 IO 则相反，没有数据时会立即返回 `EAGAIN`，线程不会睡眠。但如果程序只是不断轮询所有 fd 调 `read()`，CPU 会被空转消耗掉。

IO 多路复用的价值就在这里。它不是改变 fd 的读写语义，而是提供一种高效等待机制，让应用先把大量 fd 注册给内核，再由内核告诉应用“哪些 fd 现在已经就绪”，这样应用只处理真正有事件发生的连接。可以理解为，阻塞 IO 是“线程等一个 fd”，多路复用是“内核帮你等很多 fd，再把结果批量通知回来”。

## 题目 3：`select`、`poll`、`epoll` 有什么区别？

高级工程师回答：

`select` 是早期方案，存在 fd 数量上限，而且每次调用都要把整个 fd 集合从用户态拷到内核，再把结果返回回来，应用层还要遍历整个集合找谁就绪。`poll` 去掉了固定 fd 上限，但整体思想没有本质变化，仍然偏向“把一大批 fd 交给内核后再线性扫描”。

`epoll` 的关键改进在于把“注册关注哪些 fd”和“等待哪些 fd 就绪”拆开了。应用先通过 `epoll_ctl` 注册关注集合，之后调用 `epoll_wait` 时，内核直接返回已经就绪的事件列表，而不是让应用每次都从头扫描全部 fd。它的优势不是一句抽象的“更快”，而是更适合大规模连接、低活跃比例的事件驱动场景。

## 题目 4：为什么说 `epoll` 更适合高并发场景？

高级工程师回答：

因为高并发互联网服务的典型特征是连接很多，但任意时刻真正活跃的连接比例通常不高。如果仍然用 `select/poll` 这种每次都遍历一大批 fd 的方式，连接规模一上来，很多 CPU 时间就浪费在线性扫描和重复拷贝上。

`epoll` 更适合这类场景，是因为它让内核维护关注集合，并把就绪事件直接返回给应用。应用只处理活跃连接，不需要每次都从成千上万个连接里重新找一遍谁有事件。这让它非常适合 Nginx、Redis、网关、RPC 服务这类事件驱动架构。

## 题目 5：水平触发 LT 和边缘触发 ET 有什么区别？

高级工程师回答：

LT 的语义更直观，只要 fd 还处于可读或可写状态，内核就可能持续通知你，所以程序相对不容易漏事件。ET 更激进，它更强调“状态变化”而不是“状态持续存在”，也就是说只有当 fd 从不可用变成可用时才重点通知你一次。这样通知次数可能更少，但程序必须在一次处理里尽可能把数据读空或写尽，直到返回 `EAGAIN`。

高级工程师回答这题时，重点不应该是背定义，而是点出工程风险：ET 模式如果没有把 socket 设为非阻塞、没有循环读到 `EAGAIN`，就非常容易丢事件或把连接卡住。所以 ET 不是天然比 LT 更“高级”，而是对实现正确性要求更高。

## 题目 6：为什么 `epoll` 通常和非阻塞 socket 一起使用？

高级工程师回答：

因为 `epoll` 只负责告诉你哪个 fd 现在可能可读、可写，但如果你在处理这个 fd 时使用的是阻塞 IO，那么你仍可能在一次 `read` 或 `write` 调用上卡住线程，导致整个事件循环被拖慢。这样就失去了事件驱动模型“少量线程高效处理大量连接”的意义。

所以成熟做法通常是把 fd 设成非阻塞，由 `epoll_wait` 返回就绪事件后，再在应用层循环读写直到遇到 `EAGAIN`。这样事件循环不会因为某一个连接的一次系统调用而长期阻塞。

## 题目 7：用了 `epoll` 就一定性能好吗？

高级工程师回答：

不一定。`epoll` 解决的是“如何高效等待很多网络事件”的问题，它不解决业务处理本身慢的问题。如果你的 handler 里有重 CPU 计算、慢 SQL、慢 RPC、严重锁竞争、频繁 GC 或内存抖动，那么即使网络层已经是 `epoll`，系统整体吞吐依然可能上不去。

所以一个成熟工程师不会把 `epoll` 神化成万能优化，而会把它放在完整性能链路里看待：网络事件处理只是入口层优化，真正的瓶颈还可能在协议解析、业务逻辑、缓存命中率、下游依赖甚至机器资源隔离上。

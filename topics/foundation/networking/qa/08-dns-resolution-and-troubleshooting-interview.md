# 网络面试题 08：DNS 解析与排障

## 题目 1：一次 DNS 解析大致经历了什么过程？

高级工程师回答：

从应用视角看，通常只是发起了一次域名查询，但实际链路往往分很多层。应用会先查进程内缓存、运行时缓存、系统缓存和 `/etc/hosts`，如果都没有，再把查询交给本机配置的递归解析器。递归解析器如果缓存也未命中，才会继续代表客户端向外执行迭代查询，依次经过根 DNS、顶级域 DNS 和权威 DNS，最后拿到记录结果返回给客户端，并按 TTL 进行缓存。

高级工程师回答这题时，重点不是把根服务器背出来，而是要点出“客户端通常面对的是递归查询，递归解析器对外做的是迭代查询”，以及多层缓存会显著影响实际行为。

## 题目 2：递归查询和迭代查询有什么区别？

高级工程师回答：

递归查询的语义是“你帮我拿到最终答案”，客户端把责任交给递归解析器；而迭代查询的语义是“如果你不知道最终答案，就告诉我下一步去哪问”。在 DNS 体系里，终端应用通常只做递归查询，而递归解析器再去做多轮迭代查询。

这个区别很重要，因为很多线上问题并不是权威 DNS 自己挂了，而是递归层缓存、超时、fallback 或配置出了问题。如果你只会说“DNS 就是去根服务器查”，说明理解层次还不够。

## 题目 3：为什么改了 DNS 记录，流量没有立刻切走？

高级工程师回答：

最常见原因是缓存。你在权威 DNS 上改完记录，只代表源头数据变了，但客户端真正看到的结果取决于整条链路上的缓存是否已经过期，包括应用运行时缓存、OS 缓存、本地 DNS、递归 DNS，甚至某些代理组件自己的缓存。只要其中某一层还没失效，流量就可能继续命中旧地址。

所以高级工程师回答这题时，不会只说“等 TTL”。更完整的说法是：TTL 是理论最小边界，但真实生效时间往往受到多层缓存和组件刷新策略共同影响。

## 题目 4：为什么有时只有部分机器解析失败？

高级工程师回答：

这种情况通常说明问题不是“全局 DNS 一定挂了”，而是不同机器走了不同解析路径。可能是某批机器用的递归 DNS 配置不同，某些节点 hosts 文件被污染，某些环境开启了 IPv6 优先而另一批没有，或者运行时 DNS 缓存刷新策略不同。

所以这类问题排查时，关键不是先盯着权威记录，而是先比较不同机器实际走的解析路径、看到的 nameserver、缓存状态和最终返回结果。高级工程师更看重“差异定位能力”，而不是单点知识。

## 题目 5：DNS 慢一般慢在哪？

高级工程师回答：

DNS 慢不一定是权威 DNS 本身慢。常见瓶颈包括本地递归解析器负载高、缓存未命中导致多跳迭代查询、网络链路抖动、某个上游权威 DNS 响应慢、查询超时后切换备用 nameserver、以及双栈环境下 AAAA/A 查询策略不当。

所以一个成熟工程师不会把“连接建立慢”直接归因到服务端，他会先确认 DNS 阶段耗时有没有异常。尤其在没有阶段化观测的系统里，DNS 耗时特别容易被误记成“接口慢”。

## 题目 6：TTL 设计为什么是个工程权衡？

高级工程师回答：

TTL 太长，会让切流、迁移、容灾恢复和故障实例下线变慢；TTL 太短，又会增加解析流量、递归 DNS 压力和客户端对 DNS 链路稳定性的依赖。所以 TTL 不是越短越先进，也不是越长越稳定，而是一个切换灵活性和解析成本之间的折中。

高级工程师回答这题时，最好把 TTL 和真实业务场景绑在一起，比如灰度发布、故障切换、跨机房迁移，这样就不是在背参数，而是在讲系统设计。

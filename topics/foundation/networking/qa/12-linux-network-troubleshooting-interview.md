# 网络面试题 12：Linux 网络排障

## 题目 1：线上服务 RT 突然抖高，但 CPU 不高，你怎么排查？

高级工程师回答：

我不会先假设是业务代码慢，因为 CPU 不高通常说明系统不一定在忙算，很可能在等。我的第一步是把 RT 拆成阶段：DNS、TCP 建连、TLS 握手、连接池等待、请求发送、服务端处理、响应返回，确认到底是哪一段变长了。与此同时，我会看连接池等待时间、活跃连接数、线程或 goroutine 阻塞情况、下游 RT 和错误率，判断是不是资源在等待某个慢依赖。

如果这些信号显示连接数上升、连接池等待增长、下游 RT 变差，而 CPU 仍不高，我会优先怀疑下游变慢、连接占用拉长、重试叠加或线程在阻塞等待，而不是盲目归因到本机算力不足。高级工程师回答这题的关键是分阶段定位，而不是上来就抓包或上来就调参数。

## 题目 2：连接数暴涨，但 QPS 没涨，你会想到什么？

高级工程师回答：

这通常说明单位请求占用连接的时间变长了，或者连接回收复用出了问题，而不是单纯流量上升。我会先看长连接复用是否失效、下游 RT 是否变慢、连接池是否存在泄漏、代理和客户端是否频繁重建连接，以及 idle timeout 是否发生了不一致。因为只要连接生命周期被拉长，即使 QPS 不变，系统里同时存在的连接数也会显著增加。

如果再结合 TCP 状态分布，比如大量 `TIME_WAIT`、`CLOSE_WAIT` 或建立中的连接异常增多，就能进一步判断是短连接风暴、连接泄漏还是关闭路径异常。高级工程师要把“连接数是结果”这件事想清楚。

## 题目 3：大量 `CLOSE_WAIT` 说明什么？

高级工程师回答：

通常说明对端已经关闭连接了，本端内核也确认了，但应用层还没有把这个连接真正 close 掉。也就是说，它往往更像应用资源回收问题，而不是网络本身的问题。最典型的怀疑对象就是异常路径没释放连接、某些逻辑忘记 close、连接池或框架层归还逻辑有 bug。

排查时我会同时看 fd 数量、连接状态、代码里的借出归还路径、异常分支和慢请求栈。如果一个工程师看到 `CLOSE_WAIT` 只会说“网络有问题”，那基本说明经验还不够。

## 题目 4：什么时候你会去抓包？

高级工程师回答：

当分阶段指标和应用日志已经提示问题大概率在网络交互层，但还不足以确认具体是握手失败、RST、重传、超时还是协议行为异常时，我会去抓包。抓包不是第一步，因为它成本高、信息量大，而且如果没有前置假设，很容易淹没在噪声里。它更适合作为验证手段，而不是无方向探索手段。

例如，如果我已经发现只有某一批机器建连慢，或者怀疑连接被中间设备重置，那么抓包能直接看 SYN/SYN-ACK/ACK、RST、重传时序和 TLS 行为，这时就很有价值。

## 题目 5：内核参数调优应该在什么时候做？

高级工程师回答：

应该在你已经用指标和现象确认瓶颈点之后再做，而不是把它当默认动作。比如 `somaxconn`、fd 上限、socket buffer 等参数确实会影响接入能力和吞吐，但如果根因其实是连接泄漏、下游变慢或代理层重试风暴，那么调参数只是在掩盖问题，甚至可能把问题扩散得更远。

高级工程师不会把系统调优理解成“记住几个 sysctl 值”，而是把它当作诊断后的精细化动作。先定位，再调优，这是顺序问题。

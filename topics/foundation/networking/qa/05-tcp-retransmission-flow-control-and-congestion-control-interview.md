# 网络面试题 05：TCP 重传、流量控制与拥塞控制

## 题目 1：TCP 为什么是可靠传输？

高级工程师回答：

TCP 的可靠不是指底层网络不会丢包，而是指即使底层链路会丢包、乱序、重复，TCP 仍然尽量向应用层提供有序、可靠的字节流。它做到这一点靠的不是单一机制，而是一组协同机制，包括序列号、ACK、超时重传、快速重传、滑动窗口、流量控制和拥塞控制。

如果只回答“因为有 ACK”是不够的。ACK 只能说明对方确认到了什么位置，但真正让 TCP 在复杂网络里还能维持可靠性的，是“确认 + 重传 + 窗口 + 控制策略”这一整套组合机制。

## 题目 2：超时重传和快速重传有什么区别？

高级工程师回答：

超时重传的逻辑是：发送方发出数据后，如果在预期时间内没有收到对应确认，就推测这段数据或者 ACK 可能丢了，于是重新发送。这种方式简单有效，但缺点是恢复速度依赖超时计时器，反应可能偏慢。

快速重传则更主动一些。当发送方连续收到多个重复 ACK 时，会推测某个中间报文已经丢失，于是不等超时就直接重传缺失段。它的价值在于缩短丢包恢复时间，尤其适合单个或少量报文丢失但整体链路还在持续传输的场景。

## 题目 3：滑动窗口的作用是什么？

高级工程师回答：

滑动窗口的核心作用是提升吞吐。没有窗口机制时，发送方如果每发一个报文都停下来等 ACK，链路利用率会非常差，尤其在高 RTT 环境下更明显。滑动窗口允许发送方在没有收到全部 ACK 之前，先连续发送一批数据，只要这些数据仍处于允许的窗口范围内。

所以滑动窗口本质上是在保证可靠性的同时提升链路利用率。它不是一个“为了限速而限速”的机制，而是让 TCP 可以在 ACK 尚未全部返回时保持流水线式发送。

## 题目 4：流量控制和拥塞控制有什么区别？

高级工程师回答：

流量控制关注的是接收方是否来得及接。即使网络本身没有任何问题，如果接收端读取太慢、接收缓冲快满了，发送方也必须降速，否则就会把接收端压垮。它反映的是端点之间处理能力不匹配的问题。

拥塞控制关注的则是网络路径是否来得及承受当前流量。哪怕接收端处理能力很强，如果中间链路、交换设备、路由设备已经开始拥塞，继续猛发也只会带来更多排队、丢包和重传。简单说，流量控制面向接收端，拥塞控制面向网络整体；前者防止压垮对端，后者防止压垮网络。

## 题目 5：什么是慢启动？

高级工程师回答：

慢启动不是“非常慢地线性增加发送速度”，这个名字很容易误导。它的真正目的，是在连接初期先不要贸然把网络打满，而是逐步扩大拥塞窗口，快速探测当前链路可承受的数据量。一旦出现拥塞信号，比如丢包，就再缩减窗口，进入新的调整阶段。

所以慢启动本质上是拥塞控制里的带宽探测机制，而不是一个保守到几乎不发数据的策略。高级工程师回答这题时，重点要讲出“试探网络承载能力”这个目标。

## 题目 6：出现大量重传时，你会怎么判断问题在网络还是在应用？

高级工程师回答：

我不会一看到重传就直接下结论说“网络差”。因为重传只是结果，它可能来自链路质量问题，也可能来自交换设备队列拥塞、主机网卡压力、接收端读取不及时、系统抖动、代理层缓存问题，甚至上层服务处理不过来导致回 ACK 不及时。

我通常会结合多种信号判断，比如 RTT 是否明显抖动、丢包率是否上升、吞吐是否突然下降、主机 CPU/软中断是否异常、网卡和 socket buffer 是否积压、应用线程或 goroutine 是否卡住。也就是说，要把传输层现象和主机资源、应用行为联合起来看，而不是孤立看一个重传指标。

## 题目 7：为什么高延迟链路下，即使带宽不小，吞吐也可能上不去？

高级工程师回答：

因为吞吐不只取决于带宽，还和 RTT 以及窗口利用率密切相关。如果链路时延很高，而窗口又不够大，那么发送方很快就会把窗口发满，然后只能等 ACK 慢慢回来，导致链路虽然“理论带宽大”，但实际数据流一直断断续续，利用率不高。

这也是为什么跨地域调用、大文件传输、弱网环境下，单纯看带宽数字往往会误判性能。高级工程师应该把带宽、时延、窗口和丢包恢复一起看，而不是只盯着某一个指标。

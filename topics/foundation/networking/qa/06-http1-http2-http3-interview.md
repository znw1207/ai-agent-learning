# 网络面试题 06：HTTP/1.1、HTTP/2、HTTP/3

## 题目 1：HTTP/1.1、HTTP/2、HTTP/3 的核心差异是什么？

高级工程师回答：

如果只列特性，很容易答散。更好的方式是从连接复用、请求组织方式、队头阻塞和底层传输协议几个维度来讲。HTTP/1.1 是文本协议，支持长连接，但同一连接上的请求处理仍然偏顺序化，应用层队头阻塞明显，所以客户端往往需要开多条 TCP 连接来提升并发。HTTP/2 则通过二进制分帧和 stream 多路复用，让多个请求并发复用同一条 TCP 连接，同时配合 Header 压缩降低重复头部开销。

但 HTTP/2 仍然跑在 TCP 上，因此它虽然解决了 HTTP/1.1 的应用层队头阻塞，却没有彻底解决 TCP 层的队头阻塞问题。一旦 TCP 某个关键报文丢失，整条连接上的多个 stream 仍会受到影响。HTTP/3 则进一步建立在 QUIC 之上，也就是基于 UDP 的现代传输层设计，它把可靠传输、安全和多流管理更紧密地整合起来，从而更好地缓解了传输层队头阻塞，特别适合弱网和移动网络环境。

## 题目 2：为什么说 HTTP/2 解决了一部分队头阻塞，但没有彻底解决？

高级工程师回答：

HTTP/1.1 的典型问题是应用层队头阻塞，也就是同一连接上前一个请求没处理完，后面的请求很难继续推进。HTTP/2 通过 stream 多路复用解决了这一层问题，让多个请求响应可以并发交错传输，所以应用层视角下确实前进了一大步。

但 HTTP/2 的底层仍然是单条 TCP 连接，而 TCP 是有序字节流。一旦 TCP 某个报文丢失，后续字节在交付给上层前往往要等待重传完成，因此共享这条 TCP 连接的多个 stream 都会被连带影响。所以更准确的说法是：HTTP/2 解决了应用层队头阻塞，但没有消除 TCP 层队头阻塞。

## 题目 3：HTTP/3 为什么要改成 QUIC，而不是继续基于 TCP 演进？

高级工程师回答：

核心原因在于 TCP 这套机制虽然成熟稳定，但它的语义对多流并发场景有天然限制。HTTP/2 已经把应用层复用做得比较好了，但只要底层还是单条 TCP 字节流，就仍会被 TCP 的重传和有序交付语义整体牵制。QUIC 的价值就在于它把多流管理、可靠传输、拥塞控制和安全机制按现代互联网场景重新组织，使得单个流的丢包影响不至于像 TCP 那样更容易扩散到整条连接上的所有流。

另外，QUIC 跑在 UDP 上，也让协议演进更灵活，尤其适合移动网络、链路切换和现代 CDN/边缘网络场景。所以 HTTP/3 不是简单“把 HTTP 换个端口”，而是在传输层选择上做了架构性调整。

## 题目 4：HTTP/2 一定比 HTTP/1.1 更好吗？

高级工程师回答：

从协议能力上看，HTTP/2 通常更先进，尤其在高并发、多资源复用、头部冗余较高的场景里优势明显。但成熟工程师不会把它说成绝对更好，因为协议收益最终还要看整个系统链路。比如中间代理、TLS 终止层、老旧基础设施兼容性、监控体系和调优经验，都会影响实际落地效果。

而且在某些极端丢包场景下，HTTP/2 把大量请求聚合到少量 TCP 连接上，也可能让单连接异常放大影响范围。所以实际工程决策应该基于流量模式、链路质量和基础设施支持度，而不是仅仅因为版本号更高就默认更优。

## 题目 5：HTTP/3 最大的工程价值是什么？

高级工程师回答：

HTTP/3 最大的工程价值，不是抽象地说“更快”，而是它在现代网络条件下对时延和体验更友好，尤其是在弱网、移动端、跨网络切换、多流并发这些场景中。它基于 QUIC，更好地处理了传输层队头阻塞问题，并把安全握手和连接管理做得更现代化，这会直接影响首包时延、丢包恢复体验和链路稳定性。

但真正的高级回答还会补一句：HTTP/3 的价值要结合成本一起看。它会影响网关支持、观测手段、排障方式、UDP 策略和运维体系，所以它通常应该按业务收益灰度推进，而不是一刀切全量切换。

## 题目 6：为什么很多系统即便支持 HTTP/2 或 HTTP/3，仍然还在大量使用 HTTP/1.1？

高级工程师回答：

因为协议升级从来不是单纯的技术优劣问题，还涉及生态成熟度、设备支持、代理兼容性、监控与排障成本、团队经验以及历史系统包袱。HTTP/1.1 非常成熟，问题边界清晰，工具链完备，很多内部服务调用模型也已经围绕它形成了稳定工程实践。

所以现实里往往不是“新协议完全替代旧协议”，而是按场景并存。边缘流量、浏览器访问、CDN 场景可能优先推进 HTTP/2 或 HTTP/3，而内部某些简单稳定链路继续使用 HTTP/1.1 也是合理选择。高级工程师应该体现这种分层和分场景思维。

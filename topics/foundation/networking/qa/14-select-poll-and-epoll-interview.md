# 网络面试题 14：`select`、`poll` 和 `epoll`

## 题目 1：`select`、`poll` 和 `epoll` 都是在解决什么问题？

高级工程师回答：

它们本质上都在解决同一个问题：如何让一个线程高效地等待很多 fd 的 IO 事件，而不是为每个连接单独开一个线程。因为在线上高并发场景里，真正昂贵的往往不是单次 `read()` 或 `write()`，而是大量线程、上下文切换和空等 IO 带来的资源浪费。

所以这三个 API 不该被理解成“不同版本的网络函数”，而应该被理解成三种不同成本模型的 IO 多路复用方案。高级工程师回答这题时，重点不是背函数签名，而是先把问题背景立住。

## 题目 2：`select` 的主要问题是什么？

高级工程师回答：

`select` 的问题主要有两个。第一，它通常受 `FD_SETSIZE` 这类集合大小限制，不太适合大规模 fd 管理。第二，每次调用都要把整套 fd 集合从用户态交给内核，返回后应用还要再全量扫描一遍集合，找出哪些 fd 真正就绪。这意味着随着连接数增加，很多 CPU 时间会消耗在重复提交和线性扫描上。

所以 `select` 并不是不能用，而是更适合教学、小规模连接或兼容性要求高的场景。高级工程师回答时，应该把它的问题归结到“成本模型不适合大规模连接”，而不是简单说“它老”。

## 题目 3：`poll` 比 `select` 好在哪，又差在哪？

高级工程师回答：

`poll` 相比 `select` 的直接改进，是不再依赖固定大小的位图集合，而是用 `pollfd` 数组表达关注的 fd 和事件，因此接口更自然，也摆脱了明显的固定 fd 数量限制。但它并没有从根本上改变等待模型，因为每次调用时仍然要把整批 fd 列表交给内核，返回后应用层仍然需要遍历整组结果。

所以更准确的说法是：`poll` 比 `select` 更灵活，但没有摆脱“每轮都全量提交、全量扫描”的核心开销。这也是为什么在连接规模继续扩大后，很多 Linux 高并发服务最终还是更偏向 `epoll`。

## 题目 4：为什么 `epoll` 更适合高并发服务器？

高级工程师回答：

因为 `epoll` 把“注册关注哪些 fd”和“等待哪些 fd 真正就绪”分开了。应用层只需要在 fd 新增、修改、删除时通过 `epoll_ctl()` 维护关注集合，等待事件时调用 `epoll_wait()`，内核直接返回已经就绪的事件列表，而不是让应用每轮都从全部 fd 里重新扫一遍。这让它特别适合连接很多但活跃比例不高的互联网服务。

所以 `epoll` 的价值不该被概括成一句“更快”，而应该说它更适合大量 fd、低活跃比例、事件驱动的成本模型。高级工程师回答时，把这一点说清楚，比背一堆 API 名更重要。

## 题目 5：`epoll` 的 LT 和 ET 有什么区别？

高级工程师回答：

LT，也就是 level-triggered，可以理解成“只要这个 fd 仍然处于可读或可写状态，就持续可能通知你”，所以编程相对简单，不太容易漏事件。ET，也就是 edge-triggered，更像“状态从不可用变成可用时重点通知你一次”，这要求程序通常在一次处理里尽可能把数据读空或写尽，直到遇到 `EAGAIN`。

高级工程师回答时，重点应该是工程含义而不是术语定义：ET 对正确性要求更高，如果没有非阻塞读写和完整 drain 循环，很容易漏事件或把连接卡住。所以 ET 不是默认更高级，而是更容易写错。

## 题目 6：为什么这些机制通常都和非阻塞 socket 配合使用？

高级工程师回答：

因为 IO 多路复用只负责告诉你“哪些 fd 现在可能可以操作了”，但它不保证你对某个 fd 的一次读写调用一定不会阻塞。如果你在事件循环里拿到一个就绪 fd 后又做了阻塞式读写，那么整个线程还是可能被这个单个连接卡住，其他连接的事件就都处理不了了。

所以成熟事件驱动模型通常会把 socket 设成非阻塞，然后在就绪后循环读写直到 `EAGAIN`。高级工程师回答时，应该把“事件通知”和“后续读写语义”区分开，而不是把它们混成一件事。

## 题目 7：用了 `epoll` 就一定性能好吗？

高级工程师回答：

不一定。`epoll` 解决的是等待大量网络事件的效率问题，不会自动解决业务 handler 慢、锁竞争严重、下游依赖慢、内存分配抖动、连接池打满这些问题。如果应用层处理逻辑本身很慢，那么哪怕底层已经用了 `epoll`，系统整体吞吐也可能依然上不去。

所以高级工程师不会把 `epoll` 神化成万能优化，而会把它放到完整链路里看：它是入口层和事件等待层的优化，不是所有性能问题的统一答案。

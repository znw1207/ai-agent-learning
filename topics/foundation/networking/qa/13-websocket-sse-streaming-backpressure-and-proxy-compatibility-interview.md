# 网络面试题 13：WebSocket、SSE、流式传输、背压与代理兼容性

## 题目 1：WebSocket 和 SSE 的核心区别是什么？

高级工程师回答：

最核心的区别不是“一个高级一个简单”，而是通信模型不同。WebSocket 在 HTTP 握手后升级为一个持久双向消息通道，客户端和服务端都可以主动发消息，适合聊天、协作编辑、实时控制这类双向交互场景。SSE 则仍然运行在 HTTP 语义之上，本质是服务端持续向客户端写一个 `text/event-stream` 响应，客户端持续读取，适合通知流、日志流、AI token 流这类单向推送场景。

如果我要把这个问题答得更像高级工程师，我会再补两点。第一，SSE 不需要协议升级，通常更容易复用现有 HTTP 代理和鉴权体系；第二，WebSocket 更强，但在代理兼容性、空闲超时、会话粘性和扩缩容上通常也更复杂。

## 题目 2：什么场景你会选 SSE，而不是 WebSocket？

高级工程师回答：

当需求是“服务端持续往客户端推数据，但客户端基本不需要主动实时回传控制消息”时，我会优先考虑 SSE。典型场景包括 AI 文本流式输出、实时日志、通知流、状态订阅等。它的好处是简单、基于 HTTP、浏览器接入成本低、和现有反向代理体系更容易兼容。

如果只是想把结果边生成边推给前端，SSE 往往比 WebSocket 更合适，因为它不需要引入额外双向协议状态管理。高级工程师回答时，关键是体现“按交互模型选协议”，而不是默认选功能最强的那个。

## 题目 3：流式传输里的背压是什么意思？

高级工程师回答：

背压本质上是生产速度大于消费速度时，系统如何把压力逐层传回来，避免缓冲无限膨胀。放在流式系统里，典型情况就是服务端持续生成消息，但客户端读取很慢，或者中间代理转发很慢，于是应用队列、socket 发送缓冲区、代理 buffer 开始堆积。如果不控制，最后要么内存涨上去，要么写操作阻塞拖慢事件循环，要么整条广播链路被慢消费者拖垮。

所以高级工程师谈背压时，不会把它只当成框架名词，而会把它落实到“谁在生产、谁在消费、哪一层在积压、积压后怎么限流或丢弃”这些实际工程问题上。

## 题目 4：慢消费者为什么会拖垮流式系统？

高级工程师回答：

因为如果系统为了照顾一个读得很慢的客户端而持续缓存待发送数据，那么单个连接就可能占用越来越多内存。如果这个发送逻辑又和广播或主事件循环耦合在一起，一个慢连接还会拖慢其他正常连接，导致整体延迟上升，甚至触发资源耗尽。这类问题在聊天系统、日志流、AI token 流和订阅系统里都很典型。

所以成熟设计一般会给单连接待发送缓冲设置上限，到阈值后选择限速、丢弃部分消息、合并更新，或者直接断开慢连接。高级工程师不会把“把所有数据都尽量发完”视为唯一目标，而会优先保证整体系统稳定。

## 题目 5：为什么很多流式功能本地正常，到了线上经过代理就异常？

高级工程师回答：

因为代理层经常改变你以为的传输语义。比如 SSE 或 chunked response 在业务服务里确实是一段段写出的，但代理默认可能开启 response buffering，结果把多段内容攒成一大块再发给客户端，流式效果就没了。再比如代理 idle timeout 太短，会把正常的长连接误判为空闲连接直接断掉。WebSocket 还会额外碰到 upgrade 头透传、长连接支持和会话粘性的问题。

所以高级工程师回答这题时，一定会把应用、代理、客户端三层一起看，而不是只盯业务服务本身。很多线上流式问题，真正的根因根本不在代码逻辑，而在网关或反向代理配置。

## 题目 6：AI 流式输出为什么很多系统喜欢用 SSE？

高级工程师回答：

因为 AI 流式输出大多数时候是典型的服务端单向推送：模型不断生成 token，客户端顺序展示即可。SSE 正好契合这个模式，而且浏览器支持简单，基于 HTTP，鉴权和代理复用成本低，比 WebSocket 更轻量。对于很多“只需要把 token 流出来”的系统，这已经足够。

但如果产品需求进一步发展成中途打断、双向控制、工具调用事件交互、多人协作同步等复杂实时场景，SSE 的单向语义就会开始吃紧，这时 WebSocket 或更复杂的实时协议模型会更合适。高级工程师的关键是知道什么时候简单方案够用，什么时候该升级交互模型。

## 题目 7：你会如何设计一个流式系统的背压和兼容性策略？

高级工程师回答：

我会从端到端考虑，而不是只在代码里加一个队列。第一，给单连接待发送缓冲设置明确上限，防止慢消费者无限积压。第二，把关键消息和可丢弃消息分层，必要时允许对低价值消息做降采样或覆盖。第三，对慢连接设置阈值和断开策略。第四，检查代理层是否开启 buffering、压缩、错误的超时或不兼容的 upgrade 配置。第五，让客户端、代理、服务端在空闲超时、心跳和重连策略上保持一致。

如果系统规模再大一点，我还会把消息生产和网络发送解耦，并对广播路径做隔离，避免单个慢连接或局部拥塞拖垮整个事件分发。高级工程师的重点不是“选哪个协议”，而是“让整条链路在现实网络条件下仍然稳定”。

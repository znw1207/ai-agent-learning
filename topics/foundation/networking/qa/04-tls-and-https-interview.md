# 网络面试题 04：TLS / HTTPS

## 题目 1：HTTPS 和 HTTP 的区别是什么？

高级工程师回答：

HTTP 负责定义应用层语义，比如请求方法、URL、Header、状态码和消息体；HTTPS 并不是一种全新业务协议，而是在 HTTP 下方加了一层 TLS 安全通道，所以它本质上是 HTTP over TLS。这个区别很重要，因为它意味着 HTTPS 并没有改变你的接口语义，而是改变了传输链路的安全属性。

如果从工程角度回答，HTTPS 相比 HTTP 主要多了三件事：机密性，也就是防止明文泄漏；完整性，也就是防止传输中被篡改；身份认证，也就是让客户端确认自己连到的确实是目标服务而不是中间人伪装的站点。高级工程师回答这题时，不能只说“多了加密”，因为只有加密没有认证，仍然挡不住中间人攻击。

## 题目 2：TLS 握手主要在做什么？

高级工程师回答：

TLS 握手本质上是在建立一条后续可安全传输业务数据的会话。这个过程通常要完成几件事：协商协议版本和密码套件，服务端出示证书，客户端校验证书是否合法且域名是否匹配，然后双方协商出本次会话的对称密钥，最后才开始用对称加密去承载 HTTP 或其他应用数据。

真正值得强调的是，TLS 握手的核心不是“交换几个报文”本身，而是“认证服务端身份”和“安全协商会话密钥”这两件事。很多初中级回答会把它说成“先非对称加密，再对称加密”，这个方向不算错，但如果不把证书校验和身份认证讲出来，回答就不完整。

## 题目 3：为什么 TLS 不直接用非对称加密传输所有业务数据？

高级工程师回答：

因为非对称加密成本太高，不适合大规模业务数据传输。它更适合解决的是信任建立和密钥交换问题，而不是承载长时间、大吞吐的数据流。TLS 的常见设计就是在握手阶段利用证书和密钥交换机制建立信任，然后协商出一把会话级别的对称密钥，后续业务数据再用对称加密传输。

所以要把“非对称”和“对称”各自的职责分开理解。前者主要解决安全地建立会话，后者主要解决高效地传输数据。高级工程师回答这题时，重点应该放在性能和职责分工，而不是单纯背算法名。

## 题目 4：证书的作用是什么？

高级工程师回答：

证书的核心作用不是证明一个网站“绝对可信”，而是把某个公钥和某个身份绑定起来，并由受信任 CA 对这个绑定关系做签名背书。客户端在握手中验证的本质是：这个证书是不是由我信任的 CA 链签发，证书中的域名是否匹配我当前访问的目标，证书是否在有效期内，以及是否存在吊销等异常情况。

如果没有证书体系，客户端拿到一个公钥时根本无法确认这把公钥到底属于真实服务端还是中间人。所以证书体系解决的关键问题是“公钥归属可信”，这是 TLS 能抵抗中间人攻击的基础。

## 题目 5：TLS 为什么能防中间人攻击？

高级工程师回答：

很多人会把答案简单说成“因为链路加密了”，这不够。真正抵抗中间人的关键在于服务端身份认证，而不是仅仅加密。中间人自己也可以生成一对密钥，如果客户端不校验证书来源和域名，它完全可以把自己的公钥塞给客户端，再把流量转发给真正服务端，这样链路同样是“加密”的，但已经被中间人接管了。

TLS 之所以能防这种攻击，是因为客户端会验证服务端证书是否由受信任 CA 签发，以及证书中的域名是否匹配当前访问目标。中间人很难伪造一张既被信任链接受、又域名正确匹配的证书，所以认证链条才是防 MITM 的核心。

## 题目 6：为什么 HTTPS 往往比 HTTP 慢一点？

高级工程师回答：

HTTPS 的额外成本主要来自两个方面。第一是握手开销，它通常比明文 HTTP 多一到几个往返过程，而且还涉及证书校验和密钥交换。第二是计算开销，TLS 会引入额外的加解密和完整性校验成本。对于短连接和高 QPS 场景，这些成本会更明显。

但成熟工程师不会把它总结成“HTTPS 很慢”。更准确的说法是，HTTPS 有额外成本，但这些成本可以通过 TLS 1.3、连接复用、会话复用、长连接、网关终止 TLS 等手段显著降低。协议成本要结合系统设计看，而不是静态地下结论。

## 题目 7：什么是 TLS termination？为什么很多公司把 TLS 终止在网关？

高级工程师回答：

TLS termination 指的是 TLS 握手和解密不在最终业务进程完成，而是在反向代理、负载均衡器、API 网关、Ingress 或 Service Mesh 边车上完成。这样做的好处是可以集中管理证书、统一安全策略、减轻业务服务处理握手和证书轮换的负担，同时也便于统一观测和接入控制。

但这个设计不是没有代价。如果 TLS 在网关解密后，内部链路继续明文传输，那就必须清楚内网信任边界和风险模型；如果安全要求更高，则需要在内部再叠加 mTLS。高级工程师回答时，应该体现出这类部署不是“对或错”，而是边界、成本和风险的权衡。

## 题目 8：什么是 mTLS？它适合什么场景？

高级工程师回答：

mTLS 是 mutual TLS，也就是双向 TLS 认证。普通 HTTPS 通常只有客户端验证服务端身份，而 mTLS 要求服务端也验证客户端证书，所以双方都会证明自己的身份。这在服务间调用、零信任网络、强安全隔离、内部敏感接口访问控制里很常见。

从工程上看，mTLS 的意义不仅是“再加一层加密”，更重要的是让服务到服务之间具备强身份认证能力。尤其在微服务和多租户环境中，它可以把“谁可以调用谁”从纯网络层放行提升到密码学身份层面。

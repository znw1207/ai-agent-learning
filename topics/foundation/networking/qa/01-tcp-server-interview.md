# 网络面试题 01：TCP 建连与服务端处理

## 题目 1：客户端请求到服务端时，TCP 连接是怎么建立的？服务端程序在这个过程中做了什么？

高级工程师回答：

服务端不是在收到请求后才去“现建连接”，而是启动时先调用 `socket()` 创建监听 socket，调用 `bind()` 绑定 IP 和端口，再调用 `listen()` 进入监听状态。之后客户端发起 TCP 三次握手，请求到达后主要由服务端内核协议栈处理，而不是业务代码自己处理 SYN、ACK 报文。

第一次客户端发 SYN，服务端内核收到后会为这个连接创建未完成状态并回复 `SYN+ACK`；第三次客户端回 ACK 后，连接才真正进入 established。握手完成后，这个连接会进入可被应用层接收的状态。随后业务线程通过 `accept()` 从内核取走这个已完成连接，并拿到一个新的连接 socket fd。注意，原来的监听 socket 不负责具体收发数据，它只负责接收新连接；真正读取 HTTP/RPC 请求的是 `accept()` 返回的已连接 socket。

如果我要把这个问题讲得更像高级工程师，我会再补充两点。第一，TCP 建连和应用层请求是两回事，`accept()` 成功只代表建立了可靠字节流通道，不代表已经读到了完整 HTTP 报文。第二，服务端高并发处理通常会结合非阻塞 socket 和 `epoll` 等多路复用机制，避免大量线程阻塞在连接接入阶段。

## 题目 2：`listen(backlog)` 里的 backlog 到底是什么？

高级工程师回答：

`backlog` 不能简单回答成“连接队列长度”，那样太粗。更准确地说，它是应用层传给内核的一个排队上限提示值，用来约束监听 socket 的连接排队能力。Linux 实际会结合系统参数共同决定最终效果，例如 `somaxconn`，所以它不是绝对生效的精确数字。

工程上最常见、也最实用的理解是：`backlog` 重点影响已完成三次握手但尚未被 `accept()` 取走的全连接队列容量。也就是说，如果应用层 `accept()` 不够快，或者单机瞬时连接洪峰太大，这个队列会积压，进而导致新连接接入能力下降。

如果面试官继续深挖，我会主动补充：服务端建连一般存在两个阶段的排队语义，一类是收到 SYN 但还未完成握手的半连接阶段，另一类是完成握手但应用层还没取走的全连接阶段。不同内核版本和实现细节对 `backlog` 的映射不完全一样，但从工程诊断角度，最关键的是分清“握手压力大”与“应用 accept 不及时”是两类不同问题。

## 题目 3：半连接队列和全连接队列分别是什么？为什么要区分？

高级工程师回答：

半连接队列用于保存已经收到客户端 SYN、但还没有完成三次握手的连接状态；全连接队列用于保存已经完成三次握手、但应用层还没有调用 `accept()` 取走的连接。之所以一定要区分，是因为这两个阶段的瓶颈原因、观察指标和治理手段都不同。

如果半连接队列压力大，通常意味着 SYN 洪峰、恶意攻击、网络抖动或服务端回包异常，处理重点会放在内核参数、SYN cookies、防攻击和网络链路上。如果全连接队列压力大，说明握手基本完成了，但应用层消费连接不够快，问题更可能出在 `accept()` 调度、线程模型、事件循环负载、CPU 抢占或者下游阻塞上。

高级工程师回答这题时，不应该只停留在“有两个队列”这个定义层面，而要把它和排障动作关联起来。因为线上问题不是考你会不会背术语，而是要你区分到底是接入层抗压不够，还是应用层消费不过来。

## 题目 4：`accept()` 的本质是什么？它返回的 fd 和监听 fd 有什么区别？

高级工程师回答：

`accept()` 的本质不是“发起建连”，而是“从内核拿走一个已经完成握手、可供应用层使用的连接”。它返回的是一个新的已连接 socket fd，这个 fd 与某个具体客户端一一对应，后续 `read`/`write`、`recv`/`send` 都发生在这个 fd 上。

监听 fd 则完全不同。监听 fd 只绑定在某个本地端口上，用于持续接收入站连接，它不承载具体某个客户端会话的数据流。你可以把它理解成前台叫号窗口，而 `accept()` 返回的连接 fd 才是真正接待某个客户的单独柜台。

这个区分非常重要，因为很多并发模型的设计都建立在这个事实之上。比如一个进程可以只有一个监听 fd，但同时维护成千上万个连接 fd。Go 的 net 包、Nginx、Redis、各种 RPC 框架，本质上都依赖这个模型来做并发连接管理。

## 题目 5：如果 `accept()` 很慢，会发生什么？

高级工程师回答：

如果 `accept()` 很慢，说明应用层从内核取连接的速度跟不上连接到达速度。直接后果通常是全连接队列积压。积压到一定程度后，新完成握手的连接可能无法及时进入队列，外部表现就是客户端建连变慢、连接失败率上升、偶发超时增多。

这类问题不能只从业务代码视角看，要从系统整体看。常见原因包括事件循环被耗时逻辑阻塞、线程池过小、CPU 被其他任务抢占、单机连接洪峰过高、下游阻塞把接入线程拖慢等。排查时通常会结合连接状态统计、系统调用观测、CPU profile、事件循环延迟和应用日志来判断瓶颈点。

更高级一点的回答应该再指出一个事实：如果服务端把“接连接”和“处理业务”耦合得太死，就容易在业务高峰时反过来拖垮接入能力。所以成熟服务常把接入、协议解析、业务处理、下游调用分层解耦，并通过限流、背压、超时和负载均衡来保护接入层。

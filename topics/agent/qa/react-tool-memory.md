# 面试口述稿 01：ReAct、Tool Calling、Memory（开发岗）

用途：用于 5-10 分钟口述练习，重点是工程落地表达，不是算法推导。

## 1. 开场 30 秒（可直接背）

在 Agent 系统里，我会把核心能力拆成三块：
- ReAct 负责动态决策循环
- Tool Calling 负责把能力延伸到外部系统
- Memory 负责让系统具备跨步骤和跨会话的一致性

工程上我最关注的是：稳定性、可观测性、成本和可维护性。

## 2. ReAct 怎么讲（2 分钟）

标准结构：
- Thought：模型形成下一步意图
- Action：选择并调用工具
- Observation：读取工具返回
- Loop：继续或结束

开发岗重点不是“它很聪明”，而是“它可控”：
- 每轮循环有最大步数，防止无限循环
- 每次 Action 做参数校验，避免脏调用
- 每轮都写 trace，方便事后排查

常见追问：为什么不用纯 Prompt？
- 纯 Prompt 适合简单任务
- ReAct 适合不确定、需要外部信息的任务
- 但 ReAct 需要配套治理，否则容易漂移

## 3. Tool Calling 怎么讲（2-3 分钟）

我把工具系统拆成 4 个点：

1. 工具契约（Contract）
- 明确输入 schema / 输出 schema
- 强约束字段类型和必填项

2. 调用治理（Runtime Guardrails）
- timeout、retry、circuit breaker
- 幂等键（idempotency key）防重复副作用

3. 失败分级（Error Taxonomy）
- 可重试错误：网络抖动、超时
- 不可重试错误：权限不足、参数错误
- 业务空结果：引导澄清，不盲编答案

4. 可观测性（Observability）
- 每次工具调用记录：`tool_name`、`latency_ms`、`status`、`trace_id`
- 按工具维度看错误率和成本

一句话总结：
- Tool Calling 的本质不是“能调用”，而是“调用可治理”。

## 4. Memory 怎么讲（2-3 分钟）

我一般分短期和长期：

1. 短期记忆（Short-term State）
- 存本次任务上下文，比如步骤结果、临时变量
- 要求可恢复（checkpoint）和可回放

2. 长期记忆（Long-term Memory）
- 存用户偏好、历史事实、可复用知识
- 通常检索式读取，不直接全量塞进上下文

关键工程问题：
- 写入策略：什么信息值得写入长期记忆
- 查询效率：大规模历史下的召回速度和精度
- 数据治理：隐私脱敏、权限隔离、过期策略

常见追问：如何避免旧记忆干扰新任务？
- 记忆打标签（时间、来源、置信度）
- 查询时加时间窗与任务域过滤
- 对冲突记忆做降权或仲裁

## 5. 一段可复用的“项目化回答模板”

问题：你怎么设计 Agent 的 ReAct + Tool + Memory？

回答模板：
1. 背景
- 目标是什么，约束是什么（时延、成本、成功率）

2. 方案
- ReAct 负责动态决策
- Tool 层用 schema + 超时重试 + 错误分级
- Memory 分短期状态和长期检索

3. 稳定性
- 循环上限、幂等控制、降级策略、trace 追踪

4. 结果
- 用数据说明收益：成功率提升、P99 降低、成本下降

5. 复盘
- 当前瓶颈和下一步优化计划

## 6. 自测清单（口述前看一遍）

- 我是否把“设计原因”讲清楚，而不只是讲“用了什么框架”？
- 我是否给出了失败处理和降级路径？
- 我是否提供了可量化指标？
- 我是否能解释 trade-off（质量/速度/成本）？

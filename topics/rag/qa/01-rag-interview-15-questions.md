# Agent 开发岗 RAG 面试必答 15 题

目标：不是背概念，而是达到 Agent 开发面试里“能讲清、能设计、能排障、能选型”的水平。

---

## 一、概念题

## 1. 什么是 RAG？它解决什么问题？

参考答案：
- RAG 是 Retrieval-Augmented Generation，核心思路是先从外部知识源检索相关信息，再把这些信息作为上下文交给模型生成答案。
- 它主要解决两个问题：
  - 模型参数知识过旧
  - 长尾知识和私有知识不在模型训练语料里
- 对 Agent 来说，RAG 的价值不是“看起来更高级”，而是让 Agent 在执行任务时能访问外部事实和业务知识，降低幻觉。

## 2. 一个标准的传统 RAG 链路是什么？

参考答案：
- 标准链路通常是：
  - 文档接入与清洗
  - chunking
  - embedding / index
  - retrieve
  - rerank（可选）
  - augment context
  - generate answer
  - evaluate
- 面试里要强调：RAG 不是只有“向量检索”，而是一个完整工程链路。

## 3. RAG 和 fine-tuning 的区别是什么？

参考答案：
- RAG 是在推理时把外部知识临时补给模型。
- fine-tuning 是把能力或模式固化到模型参数里。
- RAG 更适合频繁变化的知识、私有文档、权限隔离场景。
- fine-tuning 更适合稳定风格、固定格式、行为模式的优化。
- 很多真实系统是两者结合：模型做能力定制，RAG 做知识增强。

## 4. 为什么对 Agent 开发来说，RAG 重要但不是万能？

参考答案：
- 重要，因为 Agent 需要在运行时访问知识，而不能只靠参数记忆。
- 但它不是万能，因为很多问题不是“缺知识”，而是：
  - 工具调用不稳定
  - 编排逻辑有问题
  - 权限边界不清楚
  - 状态恢复做得不好
- 所以 RAG 是 Agent 的一个关键能力模块，而不是全部。

## 二、设计题

## 5. Chunking 为什么重要？chunk size 怎么选？

参考答案：
- chunking 直接影响召回质量。
- chunk 太大：噪声多，相关信息被淹没。
- chunk 太小：语义不完整，召回后上下文不够。
- 选择原则：
  - 以“语义完整单元”为优先
  - 再结合模型上下文窗口和召回目标调参
  - 常见做法是固定长度 + overlap，或按标题/段落结构切分
- 面试里不要给死值，重点讲 trade-off。

## 6. Embedding 和向量检索你应该怎么解释？

参考答案：
- Embedding 是把文本映射成向量表示。
- 检索时，query 和文档都被映射到同一向量空间，然后通过 cosine similarity 或 dot product 找最相近的文档。
- 直觉上，相近语义的文本在向量空间里距离更近。
- 对 Agent 开发来说，重点不是数学细节，而是知道它为什么比纯关键词检索更能处理语义相近表达。

## 7. 为什么很多系统不只做向量检索，而要做 hybrid retrieval？

参考答案：
- 纯向量检索擅长语义相似，但对精确关键词、实体名、ID、版本号这类内容不一定稳定。
- 关键词检索对字面命中更强，但语义泛化弱。
- hybrid retrieval 把 BM25 / keyword retrieval 和 vector retrieval 结合，通常更稳。
- 对 Agent 系统来说，这样能降低“语义相似但事实不对”或“关键词重要却没召回”的风险。

## 8. Rerank 是什么？为什么要有它？

参考答案：
- 初始召回通常是高召回、低精度。
- rerank 的目标是对候选文档重新打分，把真正最相关的几个文档提到前面。
- 可以理解为：
  - 第一步先“广撒网”
  - 第二步再“精排序”
- 如果系统直接把初始 top-k 塞给模型，常见问题是噪声太多、上下文被无关文档污染。

## 9. 传统 RAG 和 Agentic RAG 的核心区别是什么？

参考答案：
- 传统 RAG：检索流程基本由工程代码预先固定，通常检索一次。
- Agentic RAG：检索是 Agent 的工具之一，Agent 可以决定是否检索、检索几次、是否改 query、何时停止。
- 所以差别不在“有没有向量库”，而在“检索是否由运行时决策驱动”。
- Agentic RAG 更灵活，但成本更高、评估更难、失败模式更多。

## 三、排障题

## 10. 如果 RAG 效果不好，你怎么判断问题出在检索还是生成？

参考答案：
- 先拆两层：
  - 检索层：有没有召回正确证据
  - 生成层：拿到正确证据后模型是否仍然答错
- 具体排查：
  - 看 top-k 里有没有正确文档
  - 看正确文档排序是否太靠后
  - 看上下文拼接是否把有效证据截断了
  - 看模型回答是否引用了检索证据
- 如果检索对了但回答错，问题更偏生成或 prompt。
- 如果检索本身没召回到正确文档，问题更偏数据、chunk、embedding 或 query。

## 11. 常见的 RAG 失败模式有哪些？

参考答案：
- 文档没入库或索引过期
- chunk 切得不好
- embedding 模型不适合当前语料
- query 写得太差
- top-k 太小导致漏召回
- rerank 没把正确文档提上来
- 上下文过长导致有效证据被截断
- 模型没按证据回答，出现 hallucination
- 脏数据或历史版本污染检索结果

## 12. 你会怎么评估一个 RAG 系统？

参考答案：
- 至少分两层评估：
  - retrieval evaluation
  - answer evaluation
- retrieval 关注：
  - recall@k
  - hit rate
  - 是否召回 gold document
- answer 关注：
  - correctness
  - groundedness
  - 是否引用到正确证据
- 对 Agent 场景，还可以增加：
  - 工具调用成功率
  - 多轮检索次数
  - 单任务成本
  - P95 latency

## 13. 如果知识库不断变化，你怎么处理索引和版本一致性？

参考答案：
- 不能把知识库看成静态资产，要有更新机制。
- 常见做法：
  - 文档变更触发增量重建
  - 文档加版本号和更新时间
  - 检索时按版本或时间过滤
  - 旧索引延迟淘汰，避免写入和查询瞬间不一致
- 对权限敏感系统，还要考虑按租户和权限域做隔离索引或过滤。

## 四、选型题

## 14. 什么时候应该用传统 RAG，什么时候考虑 Agentic RAG？

参考答案：
- 传统 RAG 适合：
  - 文档问答
  - FAQ
  - 知识库助手
  - 检索路径固定的问题
- Agentic RAG 适合：
  - 一次检索不够
  - 需要改 query
  - 需要多知识源联合取证
  - 需要检索和其他工具协同
- 默认建议是先用传统 RAG，把链路和评估做稳，再升级到 Agentic RAG。

## 15. 长上下文模型越来越强了，为什么还需要 RAG？

参考答案：
- 长上下文不等于高质量知识访问。
- RAG 仍然有价值，因为：
  - 外部知识会不断更新
  - 权限边界不能直接塞进模型参数
  - 全量塞上下文成本高、延迟高、噪声大
  - 检索可以显式控制知识来源和范围
- 长上下文和 RAG 不是完全替代关系，很多系统会结合使用：先检索，再把高价值内容放进长上下文。

## 五、面试收束回答

如果面试官问你“作为 Agent 开发岗，你觉得 RAG 应该掌握到什么程度”，一个比较好的回答是：

- 我不需要做到 RAG 算法研究员的深度，但我需要能把 RAG 作为 Agent 的一个能力模块来设计、实现、评估和排障。
- 我至少要能讲清楚传统 RAG 的完整链路、常见失败模式、评估方法，以及传统 RAG 和 Agentic RAG 的适用场景差异。
- 对实际项目，我会优先把检索链路做稳，再决定是否引入多轮检索和 Agentic RAG。

# agentV3 在 RAG 方向的提升路线

目标：明确 RAG 在 `agentV3` 中到底该怎么用，才不是“为了用而用”。

---

## 1. 先回答：RAG 在这个项目里真的有价值吗？

有，但价值不在“读当前 page 状态”，而在“补知识层”。

当前 `agentV3` 的真实状态来源已经很明确：

- `read_app`
- `read_page`

这些工具才是 source of truth。

所以 RAG 不应该替代当前页面结构读取。

RAG 真正有价值的地方是：

- 规则检索
- 案例检索
- 业务/API/DSL 文档检索

---

## 2. 最值得先做的三类知识源

### 2.1 页面与组件规范库

包括：

- 页面布局规范
- 组件使用规范
- 命名规范
- 表单/表格/详情页设计约定

价值：

- 降低 prompt 体积
- 让规则按需加载
- 减少风格漂移

### 2.2 历史优秀页面案例库

包括：

- 类似页面的 schema
- 类似页面的 state/methods 组织方式
- 常见页面模式（列表页、详情页、表单页）

价值：

- 提高“像你们自己系统”的程度
- 让生成更贴近现有产品风格
- 比静态 example 更容易扩展

### 2.3 业务与 API / DSL 文档库

包括：

- 字段语义说明
- 接口约束
- 表达式/DSL 使用说明
- 审批流、订单流等业务约束

价值：

- 减少字段误用
- 减少业务逻辑不合理的页面生成

---

## 3. 最合理的接入方式

### Phase 1：前置知识增强型传统 RAG

流程建议：

```text
用户需求
  -> 识别任务类型
  -> 检索最相关的规范/案例/文档
  -> 拼到 system/context
  -> 进入现有 agentLoop
```

优点：

- 不破坏现有主链路
- 最容易验证收益
- 便于快速做 A/B 对比

### Phase 2：知识检索工具化

后续再考虑增加：

- `search_design_rules`
- `search_page_examples`
- `search_business_docs`

让 Agent 在运行时决定要不要继续查。

### Phase 3：有明确需求时再升级到 Agentic RAG

只有当这些场景经常出现时再升级：

- 一次检索不够
- query 需要多轮改写
- 多知识源需要联合取证
- 需要检索后再规划再执行

---

## 4. 技术选型建议

### SDK

真正开发时，建议优先：

- 官方 embedding SDK
- 官方 LLM SDK
- 向量库官方 client

不要先手搓 HTTP 接口。

### 向量数据库

如果只是先验证价值：

- 已有 Postgres 体系：优先 `pgvector`
- 想单独做检索服务：优先 `Qdrant`

不建议一上来做过重的检索基础设施。

---

## 5. 怎么验证 RAG 真的有效

最关键的不是“加了 RAG”，而是“指标有没有变好”。

建议对比：

- 首次生成成功率
- 页面结构校验失败率
- patch 回滚率
- 人工二次修改次数
- 风格一致性

如果这些指标没改善，RAG 就只是增加了复杂度。

---

## 6. 不该用 RAG 的地方

这些位置不要用 RAG：

- 当前 page 实时状态读取
- patch 写入语义控制
- validator 逻辑
- run 恢复机制
- 权限治理

这些问题本质上是 tool / state / governance 问题，不是 knowledge retrieval 问题。
